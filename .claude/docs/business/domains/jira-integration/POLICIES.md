# Jira 연동 도메인 - 서비스 정책서

> 최종 갱신: 2026-03-29

---

## poker.jira_auth_validate

### Validation 규칙

#### POL-001: Cloud 모드 필수 입력 검증
- **규칙**: Cloud 인증 모드에서는 Jira 도메인, 계정 이메일, API Token 세 가지가 모두 입력되어야만 인증 검증을 진행할 수 있다. 공백만 입력한 경우도 미입력으로 처리한다.
- **위반 시**: "모든 필드를 입력해주세요." 오류 메시지가 표시되고 다음 단계로 진행이 차단된다.
- **비즈니스 배경**: Cloud 인증은 이메일과 토큰 조합으로 Basic Auth를 구성하므로 세 값이 모두 필요하다.

#### POL-002: Server·DC 모드 필수 입력 검증
- **규칙**: Server·DC 인증 모드에서는 Jira 베이스 URL과 Personal Access Token 두 가지만 입력하면 된다. 이메일은 요구하지 않는다.
- **위반 시**: "모든 필드를 입력해주세요." 오류 메시지가 표시되고 다음 단계로 진행이 차단된다.
- **비즈니스 배경**: Server·DC는 Bearer 토큰 단독 인증이므로 이메일이 불필요하다.

#### POL-003: 프록시 API 자격증명 헤더 검증
- **규칙**: 프록시 API는 요청에 Jira 도메인과 토큰 정보가 없으면 즉시 요청을 거부한다. 모든 Jira API 호출은 반드시 이 정보를 포함해야 한다.
- **위반 시**: 요청이 거부되고 API 호출이 실패한다.
- **비즈니스 배경**: 자격증명 없이 Jira에 요청하면 어차피 실패하므로 조기에 차단한다.

#### POL-004: 인증 성공 시에만 자격증명 캐싱
- **규칙**: Jira 인증 검증이 성공했을 때만 인증 정보(인증 모드, 도메인, 이메일, 토큰)를 브라우저 로컬 저장소에 저장한다. 인증에 실패하면 저장하지 않는다.
- **위반 시**: 해당 없음 (의도된 정책)
- **비즈니스 배경**: 잘못된 자격증명이 캐싱되어 다음 방문 시 자동 입력되는 혼란을 방지한다.

### 설정값

#### POL-005: Jira API 버전 자동 분기
- **규칙**: 이메일이 제공된 경우(Cloud 모드) Jira REST API v3를 사용하고, 이메일이 없는 경우(Server·DC 모드) Jira REST API v2를 사용한다. 인증 헤더도 각각 Basic(이메일+토큰 Base64)과 Bearer(토큰 단독)로 자동 분기된다.
- **비즈니스 의미**: 사용자가 인증 모드를 선택하면 시스템이 적합한 API 버전과 인증 방식을 자동으로 결정한다.
- **변경 시 영향**: API 버전이 변경되면 응답 스키마가 달라질 수 있으므로 이슈 파싱 로직도 함께 검토해야 한다.

#### POL-006: 도메인 URL 자동 정규화
- **규칙**: 도메인 입력값이 http 또는 https로 시작하지 않으면 시스템이 자동으로 https://를 앞에 붙인다. 끝에 슬래시가 있으면 제거한다.
- **비즈니스 의미**: 사용자가 도메인 형식을 정확히 몰라도 입력 편의를 제공한다.
- **변경 시 영향**: 내부 네트워크 등 http만 지원하는 환경에서는 문제가 발생할 수 있다.

#### POL-007: 서버는 인증 정보를 보관하지 않음
- **규칙**: 서버 프록시는 Jira 인증 정보를 저장하지 않는다. 클라이언트가 매 API 요청마다 인증 정보를 요청 헤더에 포함하여 전달하며, 서버는 이를 수신하는 즉시 Jira로 전달하고 보관하지 않는다.
- **비즈니스 의미**: 서버 침해 시 자격증명 노출 위험을 최소화한다.
- **변경 시 영향**: 해당 없음 (보안 설계 원칙)

### 에러 시나리오

#### ERR-001: 모든 필드를 입력해주세요
- **메시지**: "모든 필드를 입력해주세요."
- **발생 상황**: Cloud 모드에서 도메인·이메일·토큰 중 하나 이상이 비어있거나, Server·DC 모드에서 도메인·토큰 중 하나가 비어있을 때 다음 버튼 클릭
- **사용자 영향**: Jira 인증 검증 요청이 전송되지 않고 Step 1에 머문다.
- **대응 방법**: 누락된 필드를 모두 입력한 뒤 다시 시도한다.

#### ERR-002: Jira 인증 실패
- **메시지**: "인증 실패: {HTTP 상태코드} {Jira 응답 내용}"
- **발생 상황**: Jira 인스턴스가 인증 요청을 거부하는 경우 (잘못된 토큰, 만료된 토큰, 권한 없음 등)
- **사용자 영향**: Step 1에서 오류 메시지가 표시되고, 인증 정보가 캐싱되지 않으며, Step 2로 진행할 수 없다.
- **대응 방법**: Jira 도메인, 이메일(Cloud만 해당), 토큰이 올바른지 확인하고 재입력한다. 토큰이 만료된 경우 Jira에서 새 토큰을 발급받는다.

#### ERR-003: Jira 연결 실패
- **메시지**: "연결 실패"
- **발생 상황**: 인증 검증 중 네트워크 오류 등 예외가 발생한 경우
- **사용자 영향**: Step 1에서 일반 오류 메시지가 표시된다.
- **대응 방법**: 네트워크 상태를 확인하고 재시도한다.

## poker.jira_creds_cache

### Validation 규칙

#### POL-008: 닉네임 필수 입력 검증
- **규칙**: 닉네임 입력란이 비어있으면 Step 2를 통과할 수 없다. 공백만 입력한 경우도 미입력으로 처리한다.
- **위반 시**: "닉네임을 입력해주세요." 오류 메시지가 표시되고 Step 3로 진행이 차단된다.
- **비즈니스 배경**: 닉네임은 포커 세션에서 참가자를 식별하는 유일한 표시 이름이다.

### 상태 정책

#### POL-009: 닉네임은 기존 자격증명에 병합 저장
- **규칙**: Step 2 통과 시 기존 로컬 저장소의 자격증명 데이터에 닉네임을 추가하여 재저장한다. 자격증명 전체를 교체하지 않는다. 로컬 저장소에 기존 자격증명이 없는 경우 닉네임을 별도로 저장하지 않는다.
- **전이**: Step 2 통과 → 로컬 저장소 (인증 정보 + 닉네임) 업데이트 → Step 3
- **전이 조건**: 닉네임이 입력되어 있고, 로컬 저장소에 기존 자격증명이 존재할 때
- **역전이**: 가능 — 저장된 인증 정보 삭제 시 닉네임도 함께 삭제된다.
- **영향**: 다음 방문 시 닉네임 필드도 자동으로 채워진다.

#### POL-010: 저장된 자격증명 삭제 시 폼 전체 초기화
- **규칙**: 사용자가 저장된 인증 정보 삭제 버튼을 누르면 로컬 저장소에서 자격증명이 제거되고, 폼의 모든 입력 필드(인증 모드·도메인·이메일·토큰)가 초기 상태로 돌아간다. 삭제 버튼은 저장된 자격증명이 있을 때만 화면에 표시된다.
- **전이**: 삭제 버튼 클릭 → 로컬 저장소 제거 → 폼 초기화 → 삭제 버튼 숨김
- **전이 조건**: 저장된 자격증명이 존재할 때 (삭제 버튼이 표시된 상태)
- **역전이**: 불가 — 삭제된 자격증명은 복구할 수 없다.
- **영향**: 다음 재방문 시 폼이 빈 상태로 표시된다.

### 설정값

#### POL-011: 인증 정보 평문 저장
- **규칙**: 브라우저 로컬 저장소에 보관되는 인증 정보(토큰 포함)는 암호화 없이 평문으로 저장된다.
- **비즈니스 의미**: 브라우저 개발자 도구를 통해 토큰 값을 직접 확인할 수 있다. 공용 PC나 타인과 공유하는 기기에서 사용할 경우 토큰이 노출될 수 있다.
- **변경 시 영향**: 암호화 도입 시 기존 캐시와의 호환성 처리가 필요하다.

## poker.epic_search

### Validation 규칙

#### POL-012: Epic Key 입력 필수 검증
- **규칙**: Epic Key 입력란이 비어있으면 검색 요청을 전송하지 않는다. 검색 버튼이 비활성화 상태로 표시된다.
- **위반 시**: 검색 동작이 실행되지 않고 아무 피드백이 표시되지 않는다.
- **비즈니스 배경**: 빈 키로 Jira에 요청하면 의미없는 오류가 발생한다.

#### POL-013: Epic 타입 유효성 검증
- **규칙**: 입력한 이슈 키의 타입이 Epic이 아니면 오류를 반환한다. Epic 판별 기준은 계층 레벨이 1이거나, 이슈 타입 이름이 'epic', '에픽', '큰틀' 중 하나인 경우이다.
- **위반 시**: "{입력한 키}는 Epic 타입이 아닙니다 ({실제 이슈 타입명})" 오류가 표시되고 Epic 미리보기가 나타나지 않는다.
- **비즈니스 배경**: 포커 세션의 범위는 Epic 단위로 정의된다. Story나 Task를 잘못 입력하는 실수를 방지한다.

#### POL-014: Epic Key 변경 시 이전 결과 자동 초기화
- **규칙**: Epic Key 입력값이 변경되면 이전에 표시되던 Epic 미리보기, 하위 이슈 목록, 에러 메시지가 즉시 초기화된다.
- **위반 시**: 해당 없음 (UI 정책)
- **비즈니스 배경**: 이전 검색 결과가 남아있으면 다른 Epic의 이슈 목록으로 방을 만드는 실수가 발생할 수 있다.

#### POL-015: Epic 유효성 확인 후 하위 이슈 자동 연속 조회
- **규칙**: Epic 유효성 확인이 성공하면 해당 Epic의 하위 이슈 목록을 별도 조작 없이 자동으로 이어서 조회한다.
- **위반 시**: 해당 없음 (정상 흐름)

### 에러 시나리오

#### ERR-004: Epic을 찾을 수 없음
- **메시지**: "Epic을 찾을 수 없습니다: {입력한 Epic Key}"
- **발생 상황**: 입력한 Epic Key에 해당하는 이슈가 Jira에 존재하지 않거나 접근 권한이 없을 때
- **사용자 영향**: Epic 미리보기가 표시되지 않고 방 만들기가 불가능하다.
- **대응 방법**: Epic Key가 올바른지 확인한다 (예: PROJ-42 형식). 프로젝트 키와 이슈 번호를 확인하고, 해당 Epic에 대한 접근 권한이 있는지 확인한다.

#### ERR-005: Epic 타입 불일치
- **메시지**: "{입력한 키}는 Epic 타입이 아닙니다 ({실제 타입명})"
- **발생 상황**: 입력한 이슈 키가 존재하지만 Epic이 아닌 다른 타입(Story, Task 등)일 때
- **사용자 영향**: Epic 미리보기가 표시되지 않고 방 만들기가 불가능하다.
- **대응 방법**: Jira에서 Epic 타입의 이슈 키를 확인하여 입력한다.

#### ERR-006: Epic 검색 중 오류
- **메시지**: "Epic 검색 실패" 또는 Jira에서 반환한 구체적인 오류 메시지
- **발생 상황**: Epic 또는 이슈 목록 조회 중 네트워크 오류나 Jira API 오류가 발생한 경우
- **사용자 영향**: 에러 메시지가 표시되고 방 만들기가 불가능하다.
- **대응 방법**: 입력값을 확인하고 재시도한다. 반복 실패 시 Jira 접근 권한을 확인한다.

## poker.issues_fetch

### Validation 규칙

#### POL-016: 이슈 0건 시 방 만들기 차단
- **규칙**: Epic이 조회되었더라도 하위 이슈가 0건이면 방 만들기 버튼이 비활성화된다. 이슈가 1건 이상 있어야만 방을 생성할 수 있다.
- **위반 시**: 방 만들기 버튼이 비활성화 상태로 표시된다.
- **비즈니스 배경**: 투표할 이슈가 없는 포커 세션은 의미가 없다.

### 설정값

#### POL-017: 이슈 최대 조회 건수
- **규칙**: 한 Epic의 하위 이슈를 최대 100건까지만 조회한다. 100건을 초과하는 이슈는 포커 세션에 포함되지 않는다.
- **비즈니스 의미**: 지나치게 많은 이슈를 한 세션에 넣으면 진행이 비효율적이므로 현실적인 범위로 제한한다.
- **변경 시 영향**: 값을 높이면 매우 큰 Epic을 처리할 수 있으나 API 응답 시간이 길어질 수 있다.

#### POL-018: Story Points 필드 식별
- **규칙**: 이슈의 Story Points 값은 Jira Cloud 기준 커스텀 필드(10016번)에서 읽는다. 해당 필드가 없는 이슈는 포인트 없이 포커 세션에 포함된다.
- **비즈니스 의미**: 기존에 Story Points가 입력된 이슈는 포커 세션에서 참고값으로 활용할 수 있다.
- **변경 시 영향**: 다른 커스텀 필드를 Story Points로 사용하는 Jira 인스턴스에서는 값을 가져올 수 없다.

#### POL-019: Cloud·Server·DC 이슈 조회 방식 분기
- **규칙**: Cloud는 에픽의 직접 자식 이슈를 부모-자식 관계로 조회하고, Server·DC는 Epic Link 필드와 이슈 타입(Story, Task, Bug) 필터를 조합하여 조회한다. 두 방식의 결과는 동일한 형태로 정규화된다.
- **비즈니스 의미**: 사용자는 인증 모드만 선택하면 Cloud와 Server·DC 간 조회 방식 차이를 신경 쓰지 않아도 된다.
- **변경 시 영향**: Server·DC에서 "Epic Link" 필드 이름이 다른 경우(예: 한국어 설치 환경) 이슈 조회가 실패할 수 있다.

#### POL-020: Cloud 이슈 설명 평문 자동 변환
- **규칙**: Cloud에서 가져온 이슈 설명(Atlassian Document Format)은 시스템이 평문으로 자동 변환하여 저장한다. Server·DC의 설명은 원문(평문 또는 HTML) 그대로 사용한다.
- **비즈니스 의미**: 포커 테이블에서 이슈 설명을 일관된 텍스트 형태로 표시할 수 있다.
- **변경 시 영향**: ADF에서 지원하지 않는 노드 타입(첨부파일, 일부 미디어 등)은 변환 결과에서 누락될 수 있다.

### 에러 시나리오

#### ERR-007: 이슈 목록 조회 실패
- **메시지**: "Jira error: {HTTP 상태코드} {Jira 응답 내용}"
- **발생 상황**: 이슈 목록을 가져오는 도중 Jira가 오류를 반환한 경우 (권한 부족, JQL 오류 등)
- **사용자 영향**: Epic은 확인되었으나 이슈 목록이 표시되지 않아 방 만들기가 불가능하다.
- **대응 방법**: Jira에서 해당 Epic에 대한 조회 권한이 있는지 확인한다.

#### ERR-008: 서버 내부 오류
- **메시지**: "{예외 메시지}" 또는 "Unknown error"
- **발생 상황**: Jira 서버 연결 불가, 네트워크 오류, 또는 서버 내부 처리 중 예외가 발생한 경우
- **사용자 영향**: API에서 오류 응답 반환; UI 에러 메시지 박스에 표시된다.
- **대응 방법**: 네트워크 연결 상태를 확인한다. Jira 도메인 주소가 정확한지 확인한다. 문제가 지속되면 잠시 후 재시도한다.
