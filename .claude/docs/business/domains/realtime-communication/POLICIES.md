# 실시간 통신 도메인 - 서비스 정책서
> 최종 갱신: 2026-03-29

## poker.sse_connection

### Validation 규칙

#### POL-001: 참가자 식별자와 이름 필수 제공
- **규칙**: 방에 입장할 때 참가자 식별자와 표시 이름이 반드시 포함되어야 한다.
- **위반 시**: HTTP 400 응답이 반환되고 실시간 채널이 개설되지 않아 방 입장이 차단된다.
- **비즈니스 배경**: 서버가 피어를 식별하고 방 참가자 목록을 유지하기 위해 두 정보 모두 필요하다.

### 상태 정책

#### POL-002: 방 입장 즉시 기존 참가자 목록 수신
- **규칙**: 서버는 신규 참가자가 연결되는 즉시 이미 연결된 참가자 목록(자신 제외)을 전달한다.
- **전이**: 연결 중 → 기존 참가자 목록 보유
- **전이 조건**: SSE 채널이 성공적으로 개설된 시점
- **역전이**: 불가
- **영향**: 클라이언트는 수신된 목록을 기반으로 각 참가자와 WebRTC 협상을 즉시 시작할 수 있다.

#### POL-003: 신규 참가자 진입 시 기존 참가자 전원 알림
- **규칙**: 새 참가자가 방에 진입하면 발신자 본인을 제외한 기존 참가자 전원에게 진입 알림이 전송된다.
- **전이**: 기존 참가자의 알 수 없는 상태 → 신규 참가자 인식
- **전이 조건**: 신규 참가자의 SSE 연결 수립 완료
- **역전이**: 불가
- **영향**: 기존 참가자가 신규 참가자와 WebRTC 협상을 시작할 수 있다.

### 설정값

#### POL-004: SSE 하트비트 15초 주기
- **규칙**: 서버는 모든 활성 SSE 연결에 15초마다 빈 하트비트 프레임을 전송한다. 스트림이 이미 닫혀 있으면 전송을 중단한다.
- **비즈니스 의미**: 브라우저·프록시의 유휴 연결 강제 종료를 방지하고 끊긴 연결을 빠르게 감지한다.
- **변경 시 영향**: 값을 늘리면 끊긴 연결 감지가 지연되고, 줄이면 서버 부하가 증가한다.

## poker.webrtc_negotiation

### Validation 규칙

#### POL-005: Offer 발신 주체는 항상 신규 진입 참가자
- **규칙**: WebRTC 협상에서 Offer를 먼저 생성하는 역할은 항상 신규 진입 참가자가 담당한다. 기존 참가자는 Offer를 수신한 후 Answer를 반환한다.
- **위반 시**: 양측이 동시에 Offer를 생성하면 협상이 충돌(glare)하여 연결 실패
- **비즈니스 배경**: Full Mesh 구조에서 협상 역할을 명확히 분리해 글레어 상태를 방지한다.

#### POL-006: Offer 도착 전 ICE candidate 버퍼링
- **규칙**: Offer 수신 이전에 ICE candidate가 먼저 도착한 경우, 해당 candidate를 버퍼에 저장하고 Offer 수신 완료 후 일괄 적용한다.
- **위반 시**: 원격 정보 없이 ICE candidate를 적용하면 연결 수립 실패
- **비즈니스 배경**: 네트워크 지연으로 SDP와 ICE candidate의 도착 순서가 보장되지 않는 환경을 안전하게 처리한다.

### 설정값

#### POL-007: ICE candidate 100ms 배치 전송
- **규칙**: ICE candidate는 개별 즉시 전송하지 않고 100ms 구간에 수집된 후 일괄 전송한다. ICE gathering이 완료되면 잔여 candidate를 즉시 전송한다.
- **비즈니스 의미**: 시그널링 서버와의 왕복 횟수를 줄여 연결 수립 속도를 향상시킨다.
- **변경 시 영향**: 값을 늘리면 연결 수립이 느려지고, 줄이면 서버 요청이 증가한다.

#### POL-008: STUN 서버 3개 병렬 사용
- **규칙**: P2P 연결 시도 시 외부 STUN 서버 3개를 동시에 활용해 NAT 주소를 수집한다.
- **비즈니스 의미**: 단일 STUN 서버 장애 시에도 NAT 통과 주소 수집이 가능하다.
- **변경 시 영향**: STUN 서버 가용성에 따라 NAT 통과 성공률이 달라진다.

## poker.relay_fallback

### 상태 정책

#### POL-009: P2P 연결 실패 시 8초 후 서버 릴레이 자동 전환
- **규칙**: 참가자가 1명 이상 존재하는 방에서 8초 이내에 단 하나의 P2P DataChannel도 열리지 않으면 서버 릴레이 모드로 자동 전환한다.
- **전이**: 연결 중(connecting) → 서버 릴레이(relay)
- **전이 조건**: 8초 경과 후 열린 DataChannel 없음
- **역전이**: 불가 (릴레이 모드는 세션 종료 전까지 유지)
- **영향**: 이후 모든 게임 메시지가 서버 SSE 채널을 통해 중계된다. 방화벽·NAT 제한 환경에서도 게임 진행이 가능해진다.

#### POL-010: P2P DataChannel 개통 시 릴레이 전환 타이머 즉시 취소
- **규칙**: P2P DataChannel이 성공적으로 열리면 릴레이 전환 대기 타이머를 즉시 취소한다.
- **전이**: 연결 중(connecting) → P2P(p2p)
- **전이 조건**: DataChannel open 이벤트 수신
- **역전이**: 불가 (P2P 개통 후 릴레이로 되돌아가지 않음)
- **영향**: P2P 연결에 성공하면 서버 릴레이 없이 직접 메시지를 교환한다.

#### POL-011: 릴레이 모드 전환 시 기존 PeerConnection 즉시 정리
- **규칙**: 서버 릴레이 모드로 전환되면 모든 기존 WebRTC PeerConnection을 즉시 닫고 이름 정보만 유지한다.
- **전이**: P2P 협상 중 → 릴레이 대기 상태
- **전이 조건**: 릴레이 모드 활성화 시
- **역전이**: 불가
- **영향**: 미완성 연결이 정리되어 리소스 낭비와 오류 이벤트가 방지된다.

#### POL-012: 릴레이 모드 중복 활성화 방지
- **규칙**: 릴레이 모드는 한 번 활성화되면 이후 동일한 전환 시도를 무시한다.
- **전이**: 이미 릴레이 상태 → 변경 없음
- **전이 조건**: 릴레이 모드 전환이 중복 시도될 때
- **역전이**: 해당 없음
- **영향**: 중복 정리 및 중복 알림이 발생하지 않는다.

### Validation 규칙

#### POL-013: 릴레이 모드에서 신규 참가자 진입 시 WebRTC 협상 생략
- **규칙**: 이미 릴레이 모드로 운영 중인 방에 새 참가자가 진입하면 WebRTC 협상 없이 즉시 연결 완료로 처리한다.
- **위반 시**: 불필요한 WebRTC 협상으로 연결 지연 및 자원 낭비
- **비즈니스 배경**: 릴레이 모드에서는 서버가 중계하므로 P2P 협상이 불필요하다.

#### POL-014: 릴레이 모드 전환 시 신규 진입 참가자만 상태 동기화 요청
- **규칙**: 릴레이 모드 전환 시 신규 진입 참가자만 게임 상태 동기화 요청을 보낸다. 기존 참가자는 요청을 보내지 않는다.
- **위반 시**: 기존 참가자가 동기화 요청을 중복 전송하면 호스트가 빈 응답을 받을 수 있음
- **비즈니스 배경**: 기존 참가자는 이미 최신 게임 상태를 보유하고 있어 요청이 불필요하다.

## poker.peer_disconnect

### 상태 정책

#### POL-015: 피어 퇴장 중복 알림 방지
- **규칙**: 피어가 퇴장할 때 서버는 해당 피어가 실제로 방 목록에서 제거된 경우에만 퇴장 알림을 나머지 참가자에게 전송한다. 이미 제거된 피어의 퇴장 처리는 무시한다.
- **전이**: 피어 존재 → 피어 제거 및 퇴장 알림 전송
- **전이 조건**: SSE abort 또는 sendBeacon leave POST 중 첫 번째 처리
- **역전이**: 불가
- **영향**: SSE 연결 종료와 자발적 퇴장 신호가 경합하더라도 퇴장 알림이 한 번만 전송된다.

#### POL-016: 마지막 참가자 퇴장 시 방 자동 소멸
- **규칙**: 방의 모든 참가자가 퇴장하면 서버 메모리에서 해당 방 정보가 즉시 삭제된다.
- **전이**: 방 존재 → 방 소멸
- **전이 조건**: 마지막 참가자가 방에서 제거될 때
- **역전이**: 불가 (방이 삭제된 후 재생성 불가)
- **영향**: 이후 동일한 방 ID로 접근하면 방이 존재하지 않는 것으로 응답된다.

#### POL-017: ICE 연결 실패 즉시 참가자 해제
- **규칙**: WebRTC ICE 연결이 실패 상태로 전환되면 일반 연결 상태 이벤트보다 먼저 해당 참가자를 즉시 해제 처리한다.
- **전이**: ICE 실패 → 참가자 연결 해제
- **전이 조건**: ICE 연결 상태가 'failed'로 변경될 때
- **역전이**: 불가
- **영향**: 연결 실패가 빠르게 감지되어 UI에서 참가자가 신속히 제거된다.

## poker.room_existence_check

### Validation 규칙

#### POL-018: 방 존재 여부는 활성 연결 기준
- **규칙**: 방이 존재하는지 여부는 서버 메모리에 활성 SSE 연결을 보유한 참가자가 있는지로 판단한다. 연결 중인 참가자가 없는 방은 존재하지 않는 것으로 간주된다.
- **위반 시**: 없음 (읽기 전용 확인)
- **비즈니스 배경**: 서버 재시작이나 모든 참가자 퇴장 후에는 방이 더 이상 유효하지 않음을 보장한다.

## poker.game_message_broadcast

### Validation 규칙

#### POL-019: P2P 모드에서 열린 DataChannel에만 전송
- **규칙**: P2P 모드에서 메시지를 전송할 때 DataChannel이 열린(open) 상태인 참가자에게만 전달한다.
- **위반 시**: 닫힌 채널에 전송하면 전송 오류 발생
- **비즈니스 배경**: DataChannel 상태를 확인하지 않으면 예외가 발생해 이후 메시지 전송도 차단될 수 있다.

#### POL-020: 브로드캐스트는 발신자 본인 제외
- **규칙**: 브로드캐스트 메시지는 발신자를 제외한 나머지 참가자에게만 전달된다.
- **위반 시**: 발신자가 자신이 보낸 메시지를 다시 수신해 상태 중복 적용 발생
- **비즈니스 배경**: 발신자는 이미 자신의 상태 변경을 알고 있으므로 수신 불필요하다.

#### POL-021: 닫힌 SSE 스트림 전송 오류 무시
- **규칙**: 이미 닫힌 SSE 스트림에 메시지를 전송할 때 발생하는 오류는 무시하고 나머지 참가자 전송을 계속 진행한다.
- **위반 시**: 오류 미처리 시 전체 브로드캐스트 루프가 중단될 수 있음
- **비즈니스 배경**: 네트워크 불안정으로 일부 스트림이 예기치 않게 닫히더라도 다른 참가자에게 메시지가 전달되어야 한다.

#### POL-022: 릴레이 POST 수신자 지정 여부에 따른 전달 범위
- **규칙**: 서버 릴레이를 통한 메시지 전송 시 수신자가 지정되면 해당 참가자에게만, 지정이 없으면 발신자를 제외한 전체에게 전달한다.
- **위반 시**: 개인 메시지(sync_response)가 전체에게 전달되면 불필요한 상태 갱신 발생
- **비즈니스 배경**: 게임 메시지에 따라 전체 전달(reset/reveal)과 특정 피어 전달(sync_response)이 혼재한다.

### 에러 시나리오

#### ERR-001: 실시간 채널 연결 파라미터 누락
- **메시지**: "Missing peerId or name"
- **발생 상황**: 참가자 식별자 또는 표시 이름 없이 실시간 채널 연결 요청
- **사용자 영향**: 방 입장이 차단되고 연결이 즉시 종료된다.
- **대응 방법**: 정상적인 방 URL로 다시 접근한다. 애플리케이션이 정상 동작하는 경우 일반 사용 중 발생하지 않는다.

#### ERR-002: WebRTC P2P 연결 수립 실패
- **메시지**: (내부 경고 로그, 사용자에게 노출되지 않음)
- **발생 상황**: Offer/Answer 협상 오류 또는 ICE 연결 실패로 P2P 채널을 열 수 없음
- **사용자 영향**: 즉각적인 영향 없음. 8초 경과 후 서버 릴레이 모드로 자동 전환되어 게임 진행 가능
- **대응 방법**: 사용자 조작 불필요 — 시스템이 자동으로 서버 릴레이 모드로 전환한다.

#### ERR-003: ICE candidate 처리 오류
- **메시지**: (내부 경고 로그, 사용자에게 노출되지 않음)
- **발생 상황**: 특정 ICE candidate를 적용하는 과정에서 오류 발생
- **사용자 영향**: 해당 네트워크 경로가 사용 불가. 다른 경로로 연결 재시도 가능
- **대응 방법**: 사용자 조작 불필요 — 다른 ICE candidate로 자동 재시도되거나 릴레이 모드로 전환된다.

#### ERR-004: 릴레이 메시지 전송 실패
- **메시지**: (오류 무시, 사용자에게 노출되지 않음)
- **발생 상황**: 서버 릴레이 경유 메시지 전송 중 네트워크 오류 발생
- **사용자 영향**: 해당 메시지가 전달되지 않아 일시적인 게임 상태 불일치 발생 가능
- **대응 방법**: 재시도 없음. 지속적인 문제 발생 시 방을 나갔다가 다시 입장해 상태를 재동기화한다.
