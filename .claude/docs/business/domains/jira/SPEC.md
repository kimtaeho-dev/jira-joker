# Jira 연동 도메인 - 기획서

> 최종 갱신: 2026-03-25

## 도메인 개요

Jira 연동 도메인은 사용자가 보유한 Jira 계정 정보를 통해 Jira 시스템에 안전하게 연결하고, Planning Poker 세션에서 추정할 Epic과 하위 이슈 목록을 가져오는 역할을 담당한다. Jira Cloud와 Jira Server/Data Center 두 가지 환경을 모두 지원하며, 사용자가 환경 차이를 의식하지 않고 동일한 흐름으로 이용할 수 있도록 내부에서 자동으로 처리 방식을 분기한다.

---

## 인증 검증

### 이 기능은 무엇인가

사용자가 입력한 Jira 접속 정보(도메인 주소, 이메일, API 토큰)가 유효한지 실제 Jira 서버에 확인하는 기능이다. 방 만들기 과정의 첫 번째 단계(Step 1)에서 수행되며, 인증에 성공해야만 다음 단계(닉네임 설정)로 진행할 수 있다. 한 번 입력한 인증 정보는 브라우저에 기억되어 다음 방문 시 자동으로 채워진다.

### 사용자 시나리오

1. 사용자가 Jira 연결 화면에서 Cloud 또는 Server/DC 모드를 선택한다.
2. Cloud 모드: 도메인, 이메일, API 토큰을 입력한다. Server/DC 모드: 도메인과 Personal Access Token을 입력한다.
3. "연결" 버튼을 누르면 시스템이 입력된 정보로 Jira에 접속을 시도한다.
4. 인증에 성공하면 입력 정보가 브라우저에 저장되고, 닉네임 설정 화면(Step 2)으로 자동 이동한다.
5. 인증에 실패하면 화면 하단에 오류 메시지가 표시되고, 사용자는 정보를 수정하여 다시 시도할 수 있다.

### 왜 이렇게 동작하는가

- **Cloud/Server 모드 분리**: Jira Cloud와 Server/Data Center는 인증 방식이 근본적으로 다르다. Cloud는 이메일과 API 토큰 조합을 사용하고, Server/DC는 Personal Access Token 하나만 사용한다. 사용자가 자신의 Jira 환경에 맞는 모드를 선택하도록 하여 올바른 인증 흐름을 안내한다.
- **사전 인증 검증**: Epic 조회나 이슈 조회 단계에서 인증 오류가 발생하면 사용자 경험이 나빠지므로, 가장 첫 단계에서 인증을 확인하여 이후 단계의 안정성을 보장한다.
- **도메인 주소 자동 보정**: 사용자가 "company.atlassian.net"처럼 프로토콜 없이 입력해도 자동으로 "https://"를 붙여주어 입력 편의성을 높인다.
- **인증 정보 브라우저 저장**: 매번 도메인, 이메일, 토큰을 다시 입력하는 번거로움을 줄이기 위해 성공한 인증 정보를 브라우저에 캐싱한다.

### 주요 정책

- POL-001: Cloud 모드에서 도메인, 이메일, 토큰 세 가지 모두 입력해야 한다 (상세: POLICIES.md)
- POL-002: Server/DC 모드에서 도메인과 토큰 두 가지 모두 입력해야 한다 (상세: POLICIES.md)
- POL-003: 인증 요청 시 도메인과 토큰 정보가 반드시 포함되어야 한다 (상세: POLICIES.md)
- POL-004: 이메일 입력 여부에 따라 Cloud/Server 인증 방식이 자동으로 결정된다 (상세: POLICIES.md)
- POL-005: 도메인 주소에 프로토콜이 없으면 자동으로 https://를 추가하고 끝의 슬래시를 제거한다 (상세: POLICIES.md)

### 에러 상황

- ERR-001: 인증 정보 누락 — 정상적인 화면 흐름에서는 발생하지 않으며, 비정상 접근 시에만 표시된다
- ERR-002: Jira 인증 실패 — 도메인 주소, 이메일, API 토큰이 올바른지 확인하고 재입력한다. 토큰이 만료되었을 수 있으므로 Jira 설정에서 토큰 상태를 확인한다
- ERR-006: 필수 필드 미입력 — 비어 있는 입력 필드를 모두 채운 후 다시 시도한다

### 관련 용어

- Cloud: Atlassian이 운영하는 클라우드 환경의 Jira. 이메일과 API Token으로 인증한다
- Server/DC (Data Center): 기업이 자체 서버에 설치하여 운영하는 Jira. Personal Access Token으로 인증한다
- API Token: Jira에 프로그램적으로 접근하기 위해 발급받는 인증 키. 비밀번호 대신 사용된다
- Personal Access Token (PAT): Server/DC 환경에서 사용하는 인증 토큰

## 에픽 조회

### 이 기능은 무엇인가

사용자가 입력한 Epic 키(예: PROJ-42)로 Jira에서 해당 Epic을 찾아오는 기능이다. 방 만들기의 세 번째 단계(Step 3)에서 수행되며, 입력한 키가 실제로 존재하는 Epic인지 확인한 후 해당 Epic의 제목 등 기본 정보를 화면에 표시한다. Epic 조회에 성공하면 자동으로 하위 이슈 목록 조회가 이어진다.

### 사용자 시나리오

1. 사용자가 Epic 선택 화면에서 Epic 키(예: PROJ-42)를 입력한다.
2. "검색" 버튼을 누르면 시스템이 Jira에서 해당 이슈를 조회한다.
3. 입력한 키가 존재하고 Epic 타입이 맞으면, Epic 제목이 화면에 표시되고 자동으로 하위 이슈 목록 조회가 시작된다.
4. 입력한 키가 존재하지 않거나 Epic이 아닌 다른 타입(Story, Task 등)이면, 해당 오류 메시지가 화면에 표시된다.
5. 사용자가 Epic 키 입력 필드를 수정하면, 이전 검색 결과와 이슈 목록이 즉시 초기화된다.

### 왜 이렇게 동작하는가

- **Epic 타입 사전 검증**: Planning Poker는 Epic의 하위 이슈들을 대상으로 스토리 포인트를 추정하는 세션이다. Epic이 아닌 이슈(예: 개별 Story)를 입력하면 하위 이슈를 조회할 수 없으므로, Epic 여부를 먼저 확인하여 의미 없는 이슈 조회 시도를 방지한다.
- **다국어 Epic 판별**: Jira는 언어 설정에 따라 이슈 타입 이름이 달라진다(Epic, 에픽, 큰틀 등). 이름뿐 아니라 계층 수준 정보도 함께 확인하여 어떤 언어 환경에서도 Epic을 올바르게 판별한다.
- **입력 변경 시 결과 초기화**: 이전 Epic의 이슈 목록이 화면에 남아 있으면 새로운 Epic을 검색하기 전에 잘못된 이슈 목록으로 방을 만들 수 있다. 이를 방지하기 위해 입력값이 바뀌면 이전 결과를 즉시 지운다.

### 주요 정책

- POL-006: Epic 조회 요청에는 반드시 Epic 키가 포함되어야 한다 (상세: POLICIES.md)
- POL-007: 입력한 이슈 키가 실제 Epic 타입이어야 한다. 계층 수준과 이슈 타입 이름(Epic, 에픽, 큰틀)으로 판별한다 (상세: POLICIES.md)
- POL-008: Epic 키 입력 필드가 비어 있으면 검색 버튼이 비활성화된다 (상세: POLICIES.md)
- POL-009: Epic 키 입력값이 변경되면 이전 검색 결과가 즉시 초기화된다 (상세: POLICIES.md)

### 에러 상황

- ERR-003: Epic을 찾을 수 없음 — 입력한 Epic 키가 정확한지, 해당 Jira에 실제로 존재하는 이슈인지 확인한다
- ERR-004: Epic 타입 불일치 — 입력한 키가 Story나 Task 등 Epic이 아닌 타입이므로, Jira에서 Epic 타입의 이슈 키를 확인하여 다시 입력한다
- ERR-005: Jira 서버 오류 — Jira 서버 상태를 확인하거나, 해당 프로젝트에 대한 접근 권한이 있는지 확인한다

### 관련 용어

- Epic: Jira에서 여러 하위 이슈(Story, Task, Bug)를 묶는 상위 단위. Planning Poker 세션의 출발점이 된다
- Epic 키: Jira 이슈의 고유 식별자. 프로젝트 약어와 번호로 구성된다 (예: PROJ-42)
- 계층 수준 (Hierarchy Level): Jira 이슈의 상하위 관계를 나타내는 값. Epic은 가장 높은 수준에 해당한다

## 이슈 목록 조회

### 이 기능은 무엇인가

검증된 Epic의 하위에 속하는 이슈(Story, Task, Bug 등) 목록을 Jira에서 가져오는 기능이다. Epic 조회가 성공하면 자동으로 실행되며, 가져온 이슈 목록이 Planning Poker 세션에서 스토리 포인트를 추정할 대상 티켓이 된다. 최대 100건까지 조회할 수 있다.

### 사용자 시나리오

1. Epic 조회가 성공하면 시스템이 자동으로 해당 Epic의 하위 이슈 목록을 조회한다.
2. 조회된 이슈들이 화면에 목록으로 표시된다. 각 이슈의 키, 제목 등 기본 정보를 확인할 수 있다.
3. 이슈가 1건 이상 존재하면 "방 만들기" 버튼이 활성화된다.
4. 이슈가 없으면 "방 만들기" 버튼이 비활성화 상태로 유지되며, 다른 Epic을 검색해야 한다.
5. "방 만들기" 버튼을 누르면 조회된 이슈 목록이 포커 세션의 추정 대상 티켓으로 등록된다.

### 왜 이렇게 동작하는가

- **Cloud/Server 자동 분기**: Jira Cloud와 Server/DC는 Epic과 하위 이슈의 관계를 표현하는 방식이 다르다. Cloud는 부모-자식 관계를 직접 사용하고, Server/DC는 별도의 연결 필드를 사용한다. 시스템이 사용자의 Jira 환경을 자동 판별하여 적합한 조회 방식을 선택하므로, 사용자는 환경 차이를 신경 쓸 필요가 없다.
- **최대 100건 제한**: 한 번에 너무 많은 이슈를 가져오면 응답 시간이 길어지고 세션 운영도 비효율적이다. 대부분의 Planning Poker 세션에서 100건 이내로 충분하므로 이를 상한으로 설정하였다. 100건을 초과하는 Epic에서는 최신 이슈 기준으로 100건이 반환된다.
- **이슈 0건 시 방 생성 차단**: 추정할 이슈가 없는 상태에서 포커 세션을 시작하면 아무 작업도 할 수 없는 빈 세션이 만들어진다. 이를 방지하기 위해 이슈가 최소 1건 이상일 때만 방을 만들 수 있다.

### 주요 정책

- POL-010: 이슈 목록 조회 요청에는 반드시 Epic 키가 포함되어야 한다 (상세: POLICIES.md)
- POL-011: Jira 환경(Cloud/Server)에 따라 이슈 검색 방식이 자동으로 분기된다 (상세: POLICIES.md)
- POL-012: Epic 조회와 이슈 조회가 모두 성공하고 이슈가 1건 이상일 때만 "방 만들기" 버튼이 활성화된다 (상세: POLICIES.md)
- POL-013: 한 Epic에서 최대 100건의 이슈까지 조회 가능하며, 초과 시 최신 이슈 기준으로 반환된다 (상세: POLICIES.md)

### 에러 상황

- ERR-005: Jira 서버 오류 — Jira 서버 상태를 확인하거나, 해당 프로젝트에 대한 접근 권한이 있는지 확인한다

### 관련 용어

- Story: 사용자 관점에서 기술한 기능 요구사항 단위. 스토리 포인트 추정의 주요 대상이다
- Task: 기술적 작업이나 비기능적 활동을 나타내는 이슈 타입
- Bug: 결함이나 오류를 나타내는 이슈 타입
- Story Point: 이슈의 상대적 작업량을 나타내는 추정 단위. Planning Poker를 통해 팀원들이 합의하여 결정한다
- 최대 조회 건수 (maxResults): 한 번의 조회에서 가져올 수 있는 이슈의 최대 수. 현재 100건으로 설정되어 있다
