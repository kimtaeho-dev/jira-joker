# 방 관리 도메인 - 기획서

> 최종 갱신: 2026-03-25

## 도메인 개요

방 관리 도메인은 Planning Poker 세션이 진행되는 "방"의 전체 생애주기를 담당한다. 초대 링크를 통한 참가, 호스트와 참가자의 입장 및 이탈, 방의 생성과 소멸까지 모든 과정을 관리하며, 참가자들이 안정적으로 함께 투표할 수 있는 환경을 보장한다.

---

## 방 유효성 검사

### 이 기능은 무엇인가
초대 링크를 클릭하여 방에 접근하는 새 참가자가, 해당 방이 실제로 존재하고 활성 상태인지를 자동으로 확인하는 기능이다. 방이 이미 종료되었거나 존재하지 않는 경우 "방을 찾을 수 없습니다" 안내 화면을 보여주어, 사용자가 존재하지 않는 방에 참가를 시도하는 상황을 사전에 방지한다.

### 사용자 시나리오
1. 사용자가 호스트로부터 받은 초대 링크를 클릭한다.
2. 시스템이 서버에 해당 방이 현재 존재하는지를 자동으로 확인한다.
3. 방이 존재하면 닉네임 입력 화면(참가 폼)이 표시된다.
4. 방이 존재하지 않거나 이미 종료된 경우, "방을 찾을 수 없습니다" 안내와 함께 홈으로 돌아가는 버튼이 표시된다.

### 왜 이렇게 동작하는가
- **사전 검증으로 불필요한 연결 시도 차단**: 존재하지 않는 방에 대해 실시간 연결을 시도하면 불필요한 자원 소모와 혼란이 발생하므로, 방 존재 여부를 먼저 확인하여 차단한다.
- **새로고침 시 재검증 생략**: 이미 입장한 사용자가 페이지를 새로고침하면 브라우저에 저장된 세션 정보를 기반으로 즉시 유효한 방으로 인정한다. 이미 연결 중인 사용자에게 불필요한 서버 호출을 하지 않기 위함이다.
- **연결 기반 방 생존 판정**: 방은 별도의 데이터베이스 없이, 누군가 해당 방에 연결되어 있는 한 존재하는 것으로 판정된다. 마지막 참가자가 나가면 방은 자동으로 소멸한다.

### 주요 정책
- POL-001: 링크를 통해 접근하는 신규 참가자는 반드시 서버에 방 존재 여부를 확인해야 한다 (상세: POLICIES.md)
- POL-002: 이미 입장한 사용자는 서버 확인 없이 즉시 유효한 방으로 인정된다 (상세: POLICIES.md)
- POL-003: 방은 서버에 누군가 연결되어 있을 때만 존재하는 것으로 판정된다 (상세: POLICIES.md)

### 에러 상황
- ERR-001: 방을 찾을 수 없음 -- "홈으로 돌아가기" 버튼을 클릭하여 새 방을 만들거나, 호스트에게 유효한 초대 링크를 다시 요청한다.

### 관련 용어
- Room: Planning Poker 세션이 진행되는 가상 공간 (상세: GLOSSARY.md)
- Host: 방을 만든 사람으로 방 관리 권한을 가짐 (상세: GLOSSARY.md)

## 방 참가

### 이 기능은 무엇인가
초대 링크를 통해 방에 접근한 사용자가 닉네임을 입력하고 "참여하기" 버튼을 클릭하여 방에 참가하는 기능이다. 한번 입력한 닉네임은 브라우저에 기억되어 다음 방 참가 시 자동으로 입력된다.

### 사용자 시나리오
1. 사용자가 초대 링크를 클릭하면 닉네임 입력 화면이 표시된다.
2. 이전에 참가한 적이 있으면 마지막으로 사용한 닉네임이 자동으로 입력되어 있다.
3. 닉네임을 입력(또는 확인)하고 "참여하기" 버튼을 클릭한다.
4. 시스템이 고유 ID를 자동 발급하고 방에 입장 처리한다.
5. 참가자 수가 2명 이상이면 포커 테이블이, 미만이면 대기 화면이 표시된다.

### 왜 이렇게 동작하는가
- **닉네임 자동 캐싱**: 동일 사용자가 여러 방에 반복적으로 참가할 때 매번 닉네임을 다시 입력하는 번거로움을 줄인다.
- **고유 ID 자동 발급**: 같은 닉네임을 가진 다른 사람과 구별하기 위해 시스템이 내부적으로 고유 식별자를 자동으로 부여한다. 사용자가 이 ID를 신경 쓸 필요는 없다.

### 주요 정책
- POL-004: 닉네임이 비어 있거나 공백만 입력된 경우 참여하기 버튼이 비활성화된다 (상세: POLICIES.md)
- POL-005: 참가자 고유 ID는 시스템이 자동으로 발급하며 사용자가 지정할 수 없다 (상세: POLICIES.md)
- POL-006: 입력한 닉네임은 브라우저에 자동 저장되어 다음 참가 시 자동 입력된다 (상세: POLICIES.md)

### 에러 상황
- (해당 없음 -- 닉네임 미입력 시 버튼 비활성화로 에러 자체가 발생하지 않음)

### 관련 용어
- Participant: 방에 참가한 사용자 (상세: GLOSSARY.md)
- Nickname: 방 내에서 표시되는 사용자 이름 (상세: GLOSSARY.md)

## 대기 화면

### 이 기능은 무엇인가
방에 아직 2명 이상이 모이지 않았을 때 표시되는 화면이다. 호스트에게는 다른 사람을 초대할 수 있는 링크 공유 화면을, 참가자에게는 "호스트와 연결 중..." 안내를 표시한다. 2명 이상이 모이면 자동으로 포커 테이블 화면으로 전환된다.

### 사용자 시나리오
1. 호스트가 방을 만들면 혼자이므로 대기 화면이 표시된다.
2. 호스트에게는 초대 링크를 복사하고 공유할 수 있는 UI가 보인다.
3. 참가자가 링크를 통해 접속하면 "호스트와 연결 중..." 안내를 보며 잠시 대기한다.
4. 서로 연결이 완료되어 참가자 수가 2명 이상이 되면, 양쪽 모두 자동으로 포커 테이블 화면으로 전환된다.

### 왜 이렇게 동작하는가
- **최소 인원 제한**: Planning Poker는 여러 사람이 함께 추정하는 것이 목적이므로, 혼자서는 투표가 의미 없다. 최소 2명이 되어야 게임을 시작한다.
- **역할별 화면 분리**: 호스트는 다른 사람을 초대해야 하므로 링크 공유 UI가 필요하고, 참가자는 호스트와의 연결을 기다리므로 간소한 대기 안내만 보여주는 것이 적절하다.

### 주요 정책
- POL-007: 참가자 수가 2명 미만이면 포커 테이블을 표시하지 않고 대기 화면을 보여준다 (상세: POLICIES.md)
- POL-008: 호스트와 참가자는 대기 화면에서 각각 다른 화면을 본다 (상세: POLICIES.md)
- POL-009: 초대 링크 복사 완료 시 "복사됨!" 피드백이 2초간 표시된다 (상세: POLICIES.md)

### 에러 상황
- (해당 없음 -- 대기 화면 자체는 에러 상황이 없으며, 참가자가 모이면 자동 전환됨)

### 관련 용어
- Host: 방을 만든 사람으로 초대 링크를 공유하는 역할 (상세: GLOSSARY.md)
- Participant: 초대 링크를 통해 방에 참가한 사용자 (상세: GLOSSARY.md)

## 호스트 이탈

### 이 기능은 무엇인가
호스트가 자발적으로 방을 종료하는 기능이다. 호스트가 "방 종료" 버튼을 클릭하면 확인 팝업이 표시되고, 확인 시 모든 참가자에게 방 종료를 알린 뒤 방이 닫힌다. 이 동작은 되돌릴 수 없다.

### 사용자 시나리오
1. 호스트가 화면 상단의 "방 종료" 버튼을 클릭한다.
2. "정말로 방을 종료하시겠습니까?" 확인 팝업이 표시된다.
3. "확인"을 선택하면 모든 참가자에게 방 종료 알림이 즉시 전달된다.
4. 호스트는 홈 화면으로 이동되고, 참가자들은 "방이 종료되었습니다" 오버레이 화면을 보게 된다.
5. "취소"를 선택하면 아무 일도 일어나지 않고 기존 화면이 유지된다.

### 왜 이렇게 동작하는가
- **확인 팝업 필수**: 호스트의 방 종료는 모든 참가자를 즉시 내보내는 되돌릴 수 없는 동작이다. 실수로 방을 종료하는 것을 방지하기 위해 반드시 확인 과정을 거친다.
- **즉시 전체 알림**: 호스트가 나가면 방을 관리할 사람이 없으므로, 모든 참가자에게 즉시 종료를 알려 혼란을 방지한다.

### 주요 정책
- POL-010: 호스트가 방을 종료할 때 반드시 확인 팝업을 거쳐야 한다 (상세: POLICIES.md)
- POL-011: 호스트 이탈 시 모든 참가자에게 방 종료 메시지가 전송된다 (상세: POLICIES.md)

### 에러 상황
- ERR-002: 방이 호스트에 의해 종료됨 -- 참가자는 "홈으로 돌아가기" 버튼을 클릭하여 홈으로 이동한다.

### 관련 용어
- Host: 방을 만든 사람으로 방 종료 권한을 가짐 (상세: GLOSSARY.md)
- Room Closed: 호스트에 의해 방이 영구적으로 종료된 상태 (상세: GLOSSARY.md)

## 참가자 이탈

### 이 기능은 무엇인가
호스트가 아닌 일반 참가자가 자발적으로 방을 떠나는 기능이다. "나가기" 버튼을 클릭하면 다른 참가자들에게 이탈 사실이 즉시 알려지고, 본인은 홈 화면으로 이동한다. 방 자체는 종료되지 않으며 나머지 사람들은 계속 진행할 수 있다.

### 사용자 시나리오
1. 참가자가 화면 상단의 "나가기" 버튼을 클릭한다.
2. 시스템이 다른 모든 참가자에게 이탈 사실을 즉시 알린다.
3. 해당 참가자는 홈 화면으로 이동된다.
4. 나머지 참가자들의 화면에서 이탈한 사용자가 참가자 목록에서 사라진다.
5. 진행 중이던 투표가 있었다면, 줄어든 인원 기준으로 전원 투표 완료 여부가 다시 계산된다.

### 왜 이렇게 동작하는가
- **참가자 이탈은 확인 없이 즉시 처리**: 호스트와 달리 참가자의 이탈은 다른 사용자에게 미치는 영향이 제한적이므로, 확인 팝업 없이 바로 처리한다.
- **중복 처리 방지**: 이탈 알림이 여러 경로로 동시에 도착할 수 있으므로, 이미 목록에서 제거된 참가자에 대한 중복 처리를 방지하는 안전장치가 있다.

### 주요 정책
- POL-012: 참가자가 나가기 버튼을 클릭하면 모든 참가자에게 이탈 메시지가 전송된다 (상세: POLICIES.md)
- POL-013: 동일 참가자에 대한 이탈 메시지가 중복 수신되어도 한 번만 처리된다 (상세: POLICIES.md)

### 에러 상황
- (해당 없음 -- 참가자 이탈은 별도 에러 없이 정상 처리됨)

### 관련 용어
- Participant: 방에 참가한 사용자 (상세: GLOSSARY.md)
- Leaving: 참가자가 자발적으로 방을 떠나는 행위 (상세: GLOSSARY.md)

## 호스트 재접속 보호

### 이 기능은 무엇인가
호스트의 연결이 네트워크 문제 등으로 예기치 않게 끊어졌을 때, 방을 즉시 종료하지 않고 호스트가 다시 접속할 때까지 참가자들이 기다릴 수 있게 하는 보호 기능이다. 호스트가 같은 닉네임으로 다시 접속하면 자동으로 호스트 권한이 복원된다.

### 사용자 시나리오
1. 호스트의 네트워크가 일시적으로 끊어지거나 실수로 탭을 닫는다.
2. 참가자들의 화면에 "호스트 재접속 대기 중" 오버레이가 표시된다.
3. 참가자들은 오버레이 상태에서 대기하거나, "홈으로 돌아가기"를 선택할 수 있다.
4. 호스트가 같은 닉네임으로 방에 다시 접속한다.
5. 시스템이 닉네임이 일치함을 확인하고, 호스트 권한을 자동으로 복원한다.
6. 모든 참가자에게 호스트 복원이 알려지고, 오버레이가 사라지며 정상 화면으로 돌아간다.

### 왜 이렇게 동작하는가
- **무기한 대기 정책**: 호스트의 일시적 연결 끊김은 네트워크 불안정 등 의도치 않은 상황이므로, 즉시 방을 종료하면 진행 중이던 세션이 모두 소실된다. 참가자가 스스로 판단하여 대기하거나 떠날 수 있도록 한다.
- **닉네임 기반 호스트 판별**: 호스트가 재접속하면 새로운 고유 ID가 발급되지만, 닉네임이 동일하면 원래 호스트로 인정한다. 별도의 로그인 시스템 없이도 호스트를 식별할 수 있는 경량 방식이다.
- **복원 메시지 전송 지연**: 호스트 복원 사실을 다른 참가자에게 알릴 때, 연결이 완전히 안정화되도록 잠시 대기 후 전송한다.

### 주요 정책
- POL-014: 호스트 연결이 끊기면 방을 종료하지 않고 무기한 대기 상태로 전환된다 (상세: POLICIES.md)
- POL-015: 재접속 대기 중 동일 닉네임의 참가자가 접속하면 호스트로 자동 복원된다 (상세: POLICIES.md)
- POL-016: 호스트 복원 알림은 연결 안정화를 위해 0.5초 지연 후 전송된다 (상세: POLICIES.md)

### 에러 상황
- (해당 없음 -- 호스트가 재접속하지 않는 경우 참가자가 자발적으로 홈으로 이동하여 해소됨)

### 관련 용어
- Host: 방을 만든 사람으로 재접속 시 자동 복원 대상 (상세: GLOSSARY.md)
- Host Migration: 호스트 권한이 재접속한 사용자에게 이전되는 과정 (상세: GLOSSARY.md)

## 참가자 추방

### 이 기능은 무엇인가
호스트가 특정 참가자를 방에서 강제로 내보내는 기능이다. 호스트만 이 기능을 사용할 수 있으며, 추방된 참가자는 "방에서 추방되었습니다" 오버레이를 보고 더 이상 방에 참여할 수 없다.

### 사용자 시나리오
1. 호스트가 포커 테이블 위 참가자 목록에서 추방하려는 참가자를 선택한다.
2. 시스템이 모든 참가자에게 추방 사실을 알린다.
3. 추방된 참가자에게는 "방에서 추방되었습니다" 오버레이가 표시되고, "홈으로 돌아가기" 버튼만 누를 수 있다.
4. 나머지 참가자들의 화면에서는 추방된 사용자가 참가자 목록에서 사라진다.
5. 진행 중이던 투표가 있었다면, 줄어든 인원 기준으로 전원 투표 완료 여부가 다시 계산된다.

### 왜 이렇게 동작하는가
- **호스트 전용 권한**: 방 질서 유지 책임은 방을 만든 호스트에게 있으므로, 추방 권한은 호스트만 가진다.
- **추방 후 재입장 불가**: 추방된 참가자가 다시 같은 방에 들어오는 것을 막기 위해, 오버레이에서 홈으로만 이동할 수 있도록 한다.

### 주요 정책
- POL-017: 추방 메시지를 받은 참가자는 대상 여부에 따라 다르게 처리된다 -- 대상은 추방 오버레이, 비대상은 목록에서 제거 (상세: POLICIES.md)

### 에러 상황
- ERR-003: 방에서 추방됨 -- "홈으로 돌아가기" 버튼을 클릭하여 홈으로 이동한다. 해당 방에는 재입장할 수 없다.

### 관련 용어
- Host: 추방 권한을 가진 방의 관리자 (상세: GLOSSARY.md)
- Kick: 호스트가 특정 참가자를 강제로 내보내는 행위 (상세: GLOSSARY.md)

## 방 종료 오버레이

### 이 기능은 무엇인가
호스트가 방을 종료하거나 참가자가 추방당했을 때 화면 전체를 덮는 안내 화면(오버레이)을 표시하는 기능이다. 종료 원인에 따라 다른 메시지를 보여주며, 사용자는 "홈으로 돌아가기" 버튼을 통해서만 홈으로 이동할 수 있다.

### 사용자 시나리오
1. 참가자가 방에서 활동하는 도중 호스트가 방을 종료하거나, 호스트에 의해 추방된다.
2. 화면 전체를 덮는 오버레이가 표시된다.
   - 호스트가 방을 종료한 경우: "방이 종료되었습니다" / "호스트가 방을 나갔습니다."
   - 추방된 경우: "방에서 추방되었습니다" / "호스트에 의해 추방되었습니다."
3. 오버레이가 표시된 상태에서는 포커 게임의 어떤 기능도 사용할 수 없다.
4. "홈으로 돌아가기" 버튼을 클릭하면 세션 정보가 모두 초기화되고 홈 화면으로 이동한다.

### 왜 이렇게 동작하는가
- **원인별 메시지 분기**: 방이 종료된 것과 추방당한 것은 사용자에게 완전히 다른 의미이므로, 각각 다른 메시지를 표시하여 상황을 명확히 전달한다.
- **전체 화면 차단**: 방이 이미 종료되었거나 추방된 상태에서 다른 기능을 사용하면 오류가 발생할 수 있으므로, 오버레이로 전체 화면을 가려 안전하게 홈으로 유도한다.
- **세션 정보 초기화**: 홈으로 이동할 때 기존 방의 모든 정보를 정리하여, 이후 새 방에 참가할 때 이전 세션의 잔여 데이터로 인한 문제를 방지한다.

### 주요 정책
- POL-020: 종료 원인에 따라 오버레이에 다른 제목과 내용이 표시된다 (상세: POLICIES.md)

### 에러 상황
- ERR-002: 방이 호스트에 의해 종료됨 -- "홈으로 돌아가기" 버튼을 클릭하여 홈으로 이동한다.
- ERR-003: 방에서 추방됨 -- "홈으로 돌아가기" 버튼을 클릭하여 홈으로 이동한다.

### 관련 용어
- Disconnect Reason: 방에서 분리된 원인을 나타내는 값 (상세: GLOSSARY.md)
- Overlay: 기존 화면 위에 덮이는 전체 화면 안내 (상세: GLOSSARY.md)

## 탭 닫기/새로고침 시 이탈 처리

### 이 기능은 무엇인가
사용자가 브라우저 탭을 닫거나 페이지를 새로고침할 때, 다른 참가자들에게 이탈 사실을 최대한 빠르게 알리는 기능이다. 호스트의 경우 실수로 탭을 닫는 것을 방지하기 위해 브라우저 이탈 확인 대화상자가 추가로 표시된다.

### 사용자 시나리오

**호스트인 경우:**
1. 호스트가 탭을 닫거나 새로고침을 시도한다.
2. 브라우저 기본 이탈 확인 대화상자("이 페이지를 나가시겠습니까?")가 표시된다.
3. "나가기"를 선택하면 서버와 다른 참가자 양쪽에 이탈 사실이 동시에 전달된다.
4. "머무르기"를 선택하면 아무 일도 일어나지 않고 기존 화면이 유지된다.

**참가자인 경우:**
1. 참가자가 탭을 닫거나 새로고침을 시도한다.
2. 별도 확인 없이 서버와 다른 참가자 양쪽에 이탈 사실이 동시에 전달된다.
3. 다른 참가자들의 화면에서 해당 사용자가 참가자 목록에서 사라진다.

### 왜 이렇게 동작하는가
- **호스트만 이탈 확인 표시**: 호스트의 탭 닫기는 모든 참가자에게 큰 영향(호스트 재접속 대기 상태)을 미치므로 실수를 방지한다. 참가자의 이탈은 영향이 제한적이므로 확인 없이 처리한다.
- **이중 알림 전송**: 서버 연결과 참가자 간 직접 연결 두 가지 경로로 동시에 이탈을 알려, 어느 한쪽이 실패하더라도 다른 경로로 이탈이 감지되도록 한다. 탭 닫기는 매우 짧은 시간 내에 처리되어야 하므로 이중 전송으로 신뢰성을 높인다.

### 주요 정책
- POL-018: 호스트만 탭 닫기/새로고침 시 브라우저 이탈 확인 대화상자가 표시된다 (상세: POLICIES.md)
- POL-019: 탭 닫기 시 서버와 다른 참가자에게 동시에 이탈 사실을 알린다 (상세: POLICIES.md)

### 에러 상황
- (해당 없음 -- 탭 닫기 이탈은 best-effort 방식으로 처리되며, 전송 실패 시에도 서버가 연결 끊김을 자동 감지함)

### 관련 용어
- Host: 탭 닫기 시 이탈 확인이 표시되는 대상 (상세: GLOSSARY.md)
- Beacon: 탭이 닫히는 순간에도 전송이 보장되는 서버 알림 방식 (상세: GLOSSARY.md)

## 초대 링크 공유

### 이 기능은 무엇인가
호스트가 다른 사람을 방에 초대하기 위해 현재 방의 주소(URL)를 클립보드에 복사하는 기능이다. 대기 화면의 초대 링크 공유 영역과 화면 상단의 "링크 복사" 버튼 두 곳에서 사용할 수 있다.

### 사용자 시나리오
1. 호스트가 대기 화면의 "초대 링크 복사" 버튼 또는 화면 상단의 "링크 복사" 버튼을 클릭한다.
2. 현재 방의 주소가 클립보드에 복사된다.
3. 버튼 텍스트가 "복사됨!"으로 2초간 바뀌어 복사 성공을 알린다.
4. 호스트가 복사된 링크를 메신저, 이메일 등을 통해 초대하려는 사람에게 전달한다.
5. 초대받은 사람이 링크를 클릭하면 해당 방의 참가 화면으로 이동한다.

### 왜 이렇게 동작하는가
- **클립보드 복사 방식**: 별도의 초대 시스템(이메일 초대, 알림 등) 없이 URL 복사만으로 간단하게 초대할 수 있어 사용이 편리하다.
- **복사 완료 피드백**: 클립보드 복사는 화면에 변화가 없어 성공 여부를 알기 어려우므로, 버튼 텍스트를 잠시 변경하여 시각적 피드백을 제공한다.
- **두 곳에서 제공**: 대기 화면뿐 아니라 게임 진행 중에도 추가 참가자를 초대할 수 있도록 상단 헤더에도 복사 버튼을 배치한다.

### 주요 정책
- POL-009: 복사 완료 피드백("복사됨!")은 2초간 표시된다 (상세: POLICIES.md)
- POL-021: 방 데이터는 서버 재시작 없이 유지되어 초대 링크의 유효성이 보장된다 (상세: POLICIES.md)
- POL-022: 모든 참가자가 나간 방은 자동 삭제되어 해당 초대 링크는 더 이상 유효하지 않다 (상세: POLICIES.md)

### 에러 상황
- (해당 없음 -- 클립보드 복사 실패 시 별도 에러 처리 없음. 보안 정책에 의해 복사가 차단될 수 있으나 현재 별도 대응 없음)

### 관련 용어
- Room ID: 방을 고유하게 식별하는 값으로 초대 링크의 핵심 구성 요소 (상세: GLOSSARY.md)
- Host: 초대 링크를 공유하여 다른 사용자를 방에 초대하는 역할 (상세: GLOSSARY.md)
