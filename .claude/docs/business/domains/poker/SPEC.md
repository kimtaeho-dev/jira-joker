# 포커 게임 도메인 - 기획서

> 최종 갱신: 2026-03-25

## 도메인 개요

포커 게임 도메인은 Planning Poker 세션의 핵심 흐름을 담당한다. 방 생성부터 티켓별 투표, 결과 공개, 세션 완료까지의 전체 게임 진행 과정을 관리하며, 참가자 간 실시간 상태 동기화를 통해 모든 사람이 동일한 게임 화면을 보도록 보장한다.

---

## 방 생성 위저드

### 이 기능은 무엇인가

Planning Poker 세션을 시작하기 위해 3단계 안내 화면을 거쳐 방을 만드는 기능이다. 첫 번째 단계에서 Jira 연동 정보를 입력하고, 두 번째 단계에서 세션에서 사용할 닉네임을 정하고, 세 번째 단계에서 추정할 Epic과 하위 이슈 목록을 선택하면 방이 생성된다.

### 사용자 시나리오

1. 사용자가 홈 화면에서 Jira 연동 방식(Cloud 또는 Server)을 선택하고, 해당 방식에 맞는 인증 정보를 입력한 뒤 "다음"을 누른다.
2. 시스템이 입력한 인증 정보로 Jira에 실제 접속을 시도하여 유효성을 검증한다.
3. 인증이 성공하면 닉네임 입력 화면이 표시되고, 사용자가 닉네임을 입력하고 "다음"을 누른다.
4. Epic 키 입력 화면이 표시되고, 사용자가 Epic 키(예: PROJ-42)를 입력하고 검색 버튼을 누른다.
5. 시스템이 해당 Epic의 하위 이슈(Story, Task, Bug 등) 목록을 불러와 미리보기로 보여준다.
6. 사용자가 이슈 목록을 확인한 뒤 "방 만들기"를 누르면 방이 생성되고 게임 화면으로 이동한다.

### 왜 이렇게 동작하는가

- **인증을 가장 먼저 검증하는 이유**: 잘못된 인증 정보로 방이 만들어지면 이후 모든 이슈 조회가 실패하게 된다. 방 생성 전에 미리 검증하여 불필요한 세션 생성을 방지한다.
- **Cloud와 Server를 분리하는 이유**: Jira Cloud는 이메일과 API Token 조합으로 인증하고, Jira Server/Data Center는 Personal Access Token(PAT) 하나만으로 인증한다. 인증 방식이 다르므로 사용자가 자신의 환경을 먼저 선택해야 한다.
- **Epic 단위로 이슈를 불러오는 이유**: Planning Poker는 한 번에 여러 이슈의 규모를 추정하는 활동이다. Epic 단위로 묶으면 관련 이슈를 일괄 추정할 수 있어 효율적이다.
- **인증 정보를 브라우저에 기억하는 이유**: 같은 사용자가 반복적으로 세션을 만들 때마다 인증 정보를 다시 입력하는 번거로움을 줄이기 위해 브라우저에 자동 저장한다.

### 주요 정책

- POL-001: Cloud 인증 시 도메인, 이메일, API Token 세 가지 모두 필수 (상세: POLICIES.md)
- POL-002: Server/DC 인증 시 기본 URL과 PAT 두 가지만 필수 (상세: POLICIES.md)
- POL-003: "다음" 클릭 시 실제 Jira 접속으로 인증 유효성 검증 (상세: POLICIES.md)
- POL-004: 닉네임은 빈 값 불가 (상세: POLICIES.md)
- POL-006: 입력한 이슈는 반드시 Epic 타입이어야 함 (상세: POLICIES.md)
- POL-007: Epic 하위에 이슈가 0건이면 방 생성 불가 (상세: POLICIES.md)
- POL-010: Epic 하위 이슈는 최대 100건까지 불러옴 (상세: POLICIES.md)
- POL-011: Jira 인증 정보는 브라우저에 자동 저장됨 (상세: POLICIES.md)

### 에러 상황

- ERR-001: 필수 입력란을 채우지 않음 — 빈 칸을 모두 채운 뒤 다시 시도한다.
- ERR-002: Jira 인증 실패 — 도메인, 이메일, 토큰을 재확인하고 Cloud/Server 모드 선택이 맞는지 확인한다.
- ERR-003: 닉네임 미입력 — 닉네임을 입력한 후 다시 시도한다.
- ERR-004: Epic 검색 실패 — Epic 키 형식(예: PROJ-42)을 확인하고 재시도한다.
- ERR-005: 존재하지 않는 Epic — Jira에서 정확한 Epic 키를 확인한 뒤 다시 입력한다.
- ERR-006: Epic이 아닌 이슈 타입 입력 — Epic 타입의 이슈 키를 입력한다.
- ERR-011: Epic에 하위 이슈 없음 — 해당 Epic에 이슈를 추가하거나 다른 Epic을 선택한다.

### 관련 용어

- Epic: Jira에서 여러 하위 이슈(Story, Task 등)를 묶는 상위 이슈 단위 (상세: GLOSSARY.md)
- Cloud / Server / Data Center: Jira 호스팅 환경 유형. 인증 방식이 서로 다르다 (상세: GLOSSARY.md)
- API Token: Jira 계정에서 발급하는 인증용 문자열. 비밀번호 대신 사용한다 (상세: GLOSSARY.md)
- PAT (Personal Access Token): Jira Server/DC 환경에서 사용하는 인증 토큰 (상세: GLOSSARY.md)

## 투표

### 이 기능은 무엇인가

현재 논의 중인 티켓에 대해 참가자가 카드를 선택하여 자신의 추정치를 제출하는 기능이다. Fibonacci 수열 기반 카드(1, 2, 3, 5, 8, 13, 21)와 특수 카드("?" 추정 불가, 커피 아이콘은 휴식 요청) 중 하나를 선택할 수 있다. 다른 참가자에게는 "투표했다"는 사실만 알려지고, 실제 선택한 카드 값은 전원 투표가 끝날 때까지 비공개로 유지된다.

### 사용자 시나리오

1. 화면 하단의 카드 덱에서 원하는 카드를 클릭하여 선택한다.
2. 시스템이 "이 참가자가 투표를 완료했다"는 사실을 다른 모든 참가자에게 알린다 (실제 값은 전달하지 않음).
3. 포커 테이블 위의 해당 참가자 자리에 카드 뒷면이 표시되어 투표 완료가 시각적으로 확인된다.
4. 마음이 바뀌면 다른 카드를 다시 클릭하여 변경할 수 있다. 가장 마지막에 선택한 카드가 최종 투표로 기록된다.
5. 모든 참가자가 카드를 선택하면 자동으로 투표 공개 단계로 넘어간다.

### 왜 이렇게 동작하는가

- **투표 값을 비공개로 유지하는 이유**: Planning Poker의 핵심 원칙은 독립적인 추정이다. 다른 사람의 카드 값을 먼저 보면 앵커링 효과(먼저 본 숫자에 끌리는 현상)가 발생하여 공정한 추정이 어려워진다.
- **카드 재선택을 허용하는 이유**: 투표 공개 전이라면 생각이 바뀔 수 있으므로, 공개 전까지는 자유롭게 선택을 변경할 수 있도록 한다.
- **Fibonacci 수열을 사용하는 이유**: 큰 작업일수록 불확실성이 높아지므로, 숫자 간격이 점차 벌어지는 Fibonacci 수열이 규모 추정에 적합하다. 예를 들어, 8과 13 사이에 선택지가 없으므로 "약간 더 큰지 작은지"에 대한 불필요한 논쟁을 줄인다.
- **특수 카드("?", 커피)를 제공하는 이유**: "?"는 정보 부족으로 추정이 불가능할 때, 커피 카드는 휴식이 필요하다는 의사를 표현할 때 사용한다. 숫자 추정이 아닌 의사 표시 수단이다.

### 주요 정책

- POL-013: 전원 투표 완료 + 2인 이상일 때만 카운트다운 시작 (상세: POLICIES.md)
- POL-014: 투표 중에는 실제 카드 값을 전달하지 않고 "투표 완료" 사실만 전달 (상세: POLICIES.md)
- POL-015: 같은 라운드 내 카드 재선택 허용 (상세: POLICIES.md)
- POL-016: 카드 공개 후에는 카드 선택 비활성화 (상세: POLICIES.md)
- POL-017: 카드 덱은 1, 2, 3, 5, 8, 13, 21, ?, 커피 총 9종 (상세: POLICIES.md)

### 에러 상황

- (투표 기능 자체에는 별도 에러 상황이 없다. 카드 공개 후 선택을 시도하면 자동으로 비활성화되어 차단된다.)

### 관련 용어

- Fibonacci 카드: 1, 2, 3, 5, 8, 13, 21의 숫자가 적힌 카드. 작업 규모 추정에 사용 (상세: GLOSSARY.md)
- Planning Poker: 팀원이 독립적으로 카드를 선택하여 작업 규모를 추정하는 협업 기법 (상세: GLOSSARY.md)
- 앵커링 효과: 먼저 본 숫자에 이후 판단이 끌리는 심리적 편향 (상세: GLOSSARY.md)

## 투표 공개

### 이 기능은 무엇인가

모든 참가자가 카드를 선택하면 2초간의 카운트다운 후 자동으로 모든 카드가 동시에 공개되는 기능이다. 공개 후 결과 요약(최빈값과 평균)이 테이블 중앙에 표시되며, 호스트에게는 "다시 투표"와 "다음 티켓" 버튼이 나타난다.

### 사용자 시나리오

1. 마지막 참가자가 카드를 선택하면 테이블 중앙에 "2"부터 시작하는 카운트다운이 표시된다.
2. 2초가 지나면 각 참가자가 자신의 카드 값을 자동으로 공유한다.
3. 포커 테이블 위의 모든 참가자 자리에 카드 앞면(선택한 값)이 표시된다.
4. 테이블 중앙에 Mode(최빈값: 가장 많이 선택된 값)와 Average(평균)가 결과로 표시된다.
5. 호스트에게는 결과 아래에 "다시 투표"와 "다음 티켓"(또는 마지막 티켓이면 "완료") 버튼이 표시된다.

### 왜 이렇게 동작하는가

- **2초 카운트다운을 두는 이유**: 마지막 참가자가 투표한 직후 바로 공개되면 준비가 안 된 사용자가 놀라거나 결과를 놓칠 수 있다. 2초의 짧은 대기 시간은 "곧 공개됩니다"라는 심리적 준비 시간을 제공한다.
- **카드를 선택하지 않은 채 카운트다운이 끝나면**: 해당 참가자의 카드는 자동으로 "?"(추정 불가)로 처리된다. 별도의 에러 없이 자연스럽게 진행된다.
- **Mode와 Average 두 가지 결과를 모두 보여주는 이유**: Mode는 팀 합의에 가장 가까운 값이고, Average는 전체 경향을 보여준다. 두 값이 크게 다르면 팀 내 인식 차이가 크다는 신호로, 추가 논의가 필요함을 알 수 있다.
- **특수 카드는 평균 계산에서 제외하는 이유**: "?"(추정 불가)와 커피 카드(휴식 요청)는 숫자 추정이 아닌 의사 표시이므로 수치 계산에 포함하면 평균이 왜곡된다.
- **호스트만 다음 단계를 제어하는 이유**: 한 명이 진행을 관리해야 세션이 원활하게 이루어진다. 누구나 "다음"을 누를 수 있으면 논의가 충분히 이루어지기 전에 넘어갈 위험이 있다.

### 주요 정책

- POL-018: 전원 투표 완료 후 2초 카운트다운을 거쳐 자동 공개 (상세: POLICIES.md)
- POL-019: 카운트다운 시간은 2초 (상세: POLICIES.md)
- POL-020: Mode 계산 시 미투표자 제외 (상세: POLICIES.md)
- POL-021: Average 계산 시 특수 카드(?, 커피) 제외 (상세: POLICIES.md)
- POL-022: "다시 투표"와 "다음 티켓" 버튼은 호스트만 사용 가능 (상세: POLICIES.md)

### 에러 상황

- ERR-019: 카드 미선택 상태에서 카운트다운 종료 — 자동으로 "?"로 처리되므로 별도 조치 불필요.

### 관련 용어

- Mode (최빈값): 참가자들이 가장 많이 선택한 카드 값. 팀 합의에 가장 가까운 추정치 (상세: GLOSSARY.md)
- Average (평균): 모든 숫자 카드 값의 산술 평균. 전체 경향을 파악하는 보조 지표 (상세: GLOSSARY.md)
- 호스트: 방을 생성한 사람. 게임 진행을 제어할 수 있는 권한을 가진다 (상세: GLOSSARY.md)

## 재투표

### 이 기능은 무엇인가

투표 결과가 공개된 후, 결과에 이의가 있거나 추가 논의 후 의견이 바뀌었을 때 같은 티켓에 대해 다시 투표를 진행하는 기능이다. 호스트만 재투표를 시작할 수 있으며, 모든 참가자의 투표 기록이 초기화된 뒤 처음부터 다시 카드를 선택한다.

### 사용자 시나리오

1. 투표 결과가 공개된 상태에서 호스트가 테이블 중앙의 "다시 투표" 버튼을 클릭한다.
2. 시스템이 모든 참가자의 투표 기록을 초기화하고, "다시 투표합니다"라는 메시지를 전체에 전달한다.
3. 포커 테이블 위의 모든 카드가 사라지고, 화면 하단의 카드 덱이 다시 선택 가능 상태로 활성화된다.
4. 모든 참가자가 다시 카드를 선택하여 투표한다. 이후 흐름은 최초 투표와 동일하다.

### 왜 이렇게 동작하는가

- **호스트만 재투표를 시작할 수 있는 이유**: 재투표 여부는 팀 논의 후 결정되어야 한다. 진행자(호스트)가 논의를 이끌고 적절한 시점에 재투표를 결정한다.
- **이전 투표 기록을 완전히 초기화하는 이유**: 재투표의 목적은 논의 후 새로운 관점으로 독립적으로 추정하는 것이다. 이전 선택이 남아 있으면 앵커링 효과가 지속될 수 있다.
- **재투표 횟수에 제한이 없는 이유**: Planning Poker의 목적은 팀 합의에 도달하는 것이다. 논의 후에도 의견 차이가 크면 재투표가 필요하며, 합의 도달까지 필요한 만큼 반복할 수 있어야 한다.

### 주요 정책

- POL-022: "다시 투표" 버튼은 호스트만 사용 가능, 카드 공개 상태에서만 활성화 (상세: POLICIES.md)
- POL-023: 재투표 시 모든 참가자의 투표 상태가 초기화되고 투표 단계로 복귀 (상세: POLICIES.md)

### 에러 상황

- (재투표 기능 자체에는 별도 에러 상황이 없다. 호스트 권한이 없는 참가자에게는 버튼이 표시되지 않으므로 잘못된 접근이 불가능하다.)

### 관련 용어

- 라운드: 하나의 티켓에 대한 한 번의 투표 주기. 재투표 시 같은 티켓에 대해 새 라운드가 시작된다 (상세: GLOSSARY.md)
- 호스트: 방을 생성한 사람. 재투표와 다음 티켓 진행 권한을 가진다 (상세: GLOSSARY.md)

## 다음 티켓

### 이 기능은 무엇인가

현재 티켓의 투표가 완료된 후 호스트가 결과를 확정하고 다음 티켓으로 넘어가는 기능이다. 현재 티켓의 투표 결과(최빈값, 평균, 개별 투표)는 이력으로 영구 기록되며, 마지막 티켓이면 세션 완료 화면으로 전환된다.

### 사용자 시나리오

1. 투표 결과가 공개된 상태에서 호스트가 테이블 중앙의 "다음" 버튼을 클릭한다 (마지막 티켓인 경우 "완료" 버튼).
2. 시스템이 현재 티켓의 결과(최빈값, 평균, 모든 참가자의 개별 투표값)를 완료 이력에 기록한다.
3. 다음 티켓이 있으면: 새 티켓 정보가 표시되고, 모든 참가자의 투표 상태가 초기화되어 새 투표가 시작된다.
4. 마지막 티켓이었으면: 전체 세션 결과를 요약하는 세션 완료 화면으로 전환된다.

### 왜 이렇게 동작하는가

- **결과를 이력에 영구 보관하는 이유**: 나중에 전체 세션 결과를 한눈에 확인할 수 있어야 하고, 이력은 한 번 기록되면 수정할 수 없어 결과의 신뢰성을 보장한다.
- **각 참가자의 개별 투표값도 함께 기록하는 이유**: 최빈값과 평균뿐 아니라 누가 어떤 값을 선택했는지도 기록하여, 이후 의견 차이 분석이나 회고에 활용할 수 있다.
- **호스트만 다음 티켓으로 넘길 수 있는 이유**: 결과에 대한 충분한 논의가 이루어졌는지 진행자가 판단해야 한다. 참가자가 임의로 넘길 수 있으면 논의 없이 진행될 위험이 있다.

### 주요 정책

- POL-022: "다음 티켓" 버튼은 호스트만 사용 가능, 카드 공개 상태에서만 활성화 (상세: POLICIES.md)
- POL-024: 다음 티켓 진행 시 현재 결과 기록 후 투표 단계로 전환 (상세: POLICIES.md)
- POL-025: 완료 티켓의 결과(최빈값, 평균, 개별 투표)는 불변 기록으로 보존 (상세: POLICIES.md)
- POL-026: 현재 티켓이 목록의 마지막인지 자동 판별하여 버튼 텍스트 변경 (상세: POLICIES.md)

### 에러 상황

- (다음 티켓 기능 자체에는 별도 에러 상황이 없다. 호스트 권한이 없는 참가자에게는 버튼이 표시되지 않는다.)

### 관련 용어

- CompletedTicket: 투표가 완료된 티켓의 결과 기록. 최빈값, 평균, 개별 투표를 포함한다 (상세: GLOSSARY.md)
- Story Point (SP): 작업의 상대적 규모를 나타내는 단위. 투표 결과로 결정된다 (상세: GLOSSARY.md)

## 티켓 패널

### 이 기능은 무엇인가

화면 오른쪽에 고정된 사이드 패널로, 현재 논의 중인 티켓의 상세 정보와 이미 완료된 티켓의 이력을 보여주는 기능이다. 토글 버튼으로 열거나 닫을 수 있으며, 패널이 닫히면 게임 화면이 전체 너비를 사용하고, 열리면 게임 화면이 자동으로 중앙 정렬된다.

### 사용자 시나리오

1. 화면 오른쪽 가장자리의 토글 버튼을 클릭하면 티켓 패널이 펼쳐진다.
2. 패널 상단에는 현재 티켓의 키(예: PROJ-123), 진행 상황(3/10), 제목, 상세 설명이 표시된다.
3. 패널 하단에는 이미 완료된 티켓의 이력이 접을 수 있는 목록으로 표시된다.
4. 다음 티켓으로 넘어가면 패널의 내용이 자동으로 갱신된다.
5. 토글 버튼을 다시 클릭하면 패널이 닫히고 게임 화면이 넓어진다.

### 왜 이렇게 동작하는가

- **티켓 정보를 별도 패널로 분리한 이유**: 포커 테이블과 카드 덱이 화면 중앙을 차지하므로, 티켓 상세 정보는 별도 영역으로 분리하여 투표에 집중하면서도 필요할 때 참조할 수 있도록 했다.
- **패널을 열고 닫을 수 있는 이유**: 화면이 좁은 환경에서는 패널을 닫아 게임 영역을 넓게 사용하고, 티켓 내용을 확인할 때만 열 수 있도록 유연성을 제공한다.
- **패널이 게임 상태를 변경하지 않는 이유**: 패널은 순수하게 정보 표시 용도이며, 투표나 진행에 영향을 주지 않는다. 읽기 전용으로 설계하여 실수로 게임 흐름을 방해하는 일을 방지한다.

### 주요 정책

- (티켓 패널은 읽기 전용 표시 기능이므로 별도 비즈니스 정책이 적용되지 않는다. 표시 데이터는 투표 및 다음 티켓 기능의 정책을 따른다.)

### 에러 상황

- (티켓 패널 자체에는 에러 상황이 없다. 표시할 완료 이력이 없으면 이력 영역이 표시되지 않는다.)

### 관련 용어

- TicketDetail: 현재 진행 중인 티켓의 키, 제목, 설명을 보여주는 영역 (상세: GLOSSARY.md)
- TicketHistory: 이미 투표가 완료된 티켓의 결과 목록 (상세: GLOSSARY.md)

## 세션 완료

### 이 기능은 무엇인가

모든 티켓의 투표가 완료되면 표시되는 최종 결과 화면이다. 티켓별 투표 결과(최빈값, 평균)를 테이블로 정리하고, 전체 Story Point 합산을 보여준다. 결과를 확인한 후 "세션 종료" 버튼으로 방을 나갈 수 있다.

### 사용자 시나리오

1. 마지막 티켓의 투표 공개 후 호스트가 "완료" 버튼을 클릭한다.
2. 게임 화면 대신 세션 완료 화면이 표시된다.
3. 모든 티켓이 테이블(또는 모바일에서는 카드) 형태로 나열되며, 각 티켓마다 키, 제목, 최빈값, 평균이 표시된다.
4. 테이블 하단에 총 Story Point 합산이 표시된다.
5. "세션 종료" 버튼을 클릭하면 방이 정리되고 홈 화면으로 돌아간다.

### 왜 이렇게 동작하는가

- **총 SP 합산에서 특수 카드를 제외하는 이유**: "?"(추정 불가)나 커피 카드(휴식 요청)로 결정된 티켓은 구체적인 포인트가 없으므로 합산에 포함하면 총 SP가 부정확해진다. 숫자 값으로 결정된 티켓만 합산한다.
- **세션 종료 시 방을 완전히 정리하는 이유**: 세션이 끝나면 모든 게임 상태를 초기화하고 홈 화면으로 돌려보내, 같은 방이 의도치 않게 재사용되는 것을 방지한다.
- **데스크톱과 모바일 표시를 다르게 하는 이유**: 데스크톱에서는 테이블 형태가 한눈에 보기 좋지만, 모바일의 좁은 화면에서는 카드 형태가 더 읽기 쉽다.

### 주요 정책

- POL-027: 모든 티켓 소진 + 티켓 목록이 비어있지 않을 때 세션 완료 화면 표시 (상세: POLICIES.md)
- POL-028: 총 SP 합산 시 숫자 값만 포함하고, 특수 카드로 결정된 티켓은 0으로 처리 (상세: POLICIES.md)

### 에러 상황

- (세션 완료 화면 자체에는 에러 상황이 없다. 세션 종료 버튼은 홈 화면으로의 정상 이동을 수행한다.)

### 관련 용어

- Story Point (SP): 작업의 상대적 규모 단위. 세션 완료 화면에서 전체 합산이 표시된다 (상세: GLOSSARY.md)
- Mode (최빈값): 가장 많은 참가자가 선택한 값. 세션 완료 화면의 티켓별 결과로 표시된다 (상세: GLOSSARY.md)
- Average (평균): 숫자 카드의 산술 평균. 세션 완료 화면의 티켓별 보조 결과로 표시된다 (상세: GLOSSARY.md)

## 상태 동기화

### 이 기능은 무엇인가

세션 도중에 새로 참여한 사람이 현재 진행 상황을 즉시 따라잡을 수 있도록, 기존 참가자의 게임 상태를 자동으로 전달받는 기능이다. 새 참가자가 접속하면 자동으로 "현재 상태를 알려달라"는 요청을 보내고, 기존 참가자가 현재 투표 단계, 티켓 진행 상황, 참가자 목록 등을 응답하여 동기화한다.

### 사용자 시나리오

1. 새 참가자가 초대 링크를 통해 방에 입장한다.
2. 기존 참가자와의 연결이 수립되면 시스템이 자동으로 게임 상태를 요청한다.
3. 기존 참가자 중 한 명이 현재 게임 상태(티켓 목록, 진행 중인 티켓 번호, 투표 단계, 참가자 목록, 완료 이력 등)를 응답한다.
4. 새 참가자의 화면에 현재 게임 상황이 즉시 반영된다. 예를 들어, 3번째 티켓의 투표가 진행 중이라면 그 상태로 바로 진입한다.
5. 새 참가자가 이미 카드를 선택한 상태였다면(예: 새로고침으로 재접속) 자신의 선택은 유지된다.

### 왜 이렇게 동작하는가

- **새 참가자만 상태를 요청하는 이유**: 기존 참가자가 불필요하게 상태 요청을 보내면 빈 상태가 동기화되어 진행 중인 게임이 초기화될 수 있다. 새로 합류하는 쪽만 요청하도록 제한하여 이런 위험을 방지한다.
- **자신의 투표를 보존하는 이유**: 브라우저 새로고침 등으로 재접속한 경우, 이미 선택한 카드 값이 동기화 과정에서 사라지면 다시 투표해야 하는 불편함이 생�다. 자신의 투표는 항상 로컬 값을 우선하여 보존한다.
- **중복 참가자 추가를 방지하는 이유**: 네트워크 상태에 따라 같은 사람의 접속 이벤트가 여러 번 도착할 수 있다. 동일한 참가자가 목록에 중복 표시되지 않도록 차단한다.
- **Jira 인증 정보를 동기화에서 제외하는 이유**: Jira API 토큰은 민감한 정보이므로 다른 참가자에게 전달되어서는 안 된다. 게임 진행에 필요한 데이터(티켓 목록, 투표 상태)만 동기화한다.

### 주요 정책

- POL-029: 상태 요청은 새로 합류한 참가자만 발신 (상세: POLICIES.md)
- POL-030: 동기화 수신 시 자신의 기존 투표값은 보존 (상세: POLICIES.md)
- POL-031: 동일 참가자의 중복 추가 차단 (상세: POLICIES.md)
- POL-032: 이미 퇴장한 참가자의 중복 이탈 이벤트 무시 (상세: POLICIES.md)
- POL-042: Jira 인증 정보는 동기화 데이터에 포함되지 않음 (상세: POLICIES.md)
- POL-043: 게임 상태는 브라우저 세션에 자동 저장되어 새로고침 후에도 복원됨 (상세: POLICIES.md)

### 에러 상황

- (상태 동기화는 자동으로 이루어지며 사용자에게 별도 에러가 표시되는 상황은 없다. 동기화 실패 시에도 기존 참가자와의 일반적인 투표 메시지 교환으로 점진적으로 상태가 맞춰진다.)

### 관련 용어

- 동기화 요청 (sync_request): 새 참가자가 기존 참가자에게 현재 게임 상태를 달라고 보내는 요청 (상세: GLOSSARY.md)
- 동기화 응답 (sync_response): 기존 참가자가 현재 게임 상태 전체를 담아 보내는 응답 (상세: GLOSSARY.md)
- 세션 스토리지: 브라우저 탭 단위로 게임 상태를 임시 저장하는 공간. 새로고침 시 상태 복원에 사용 (상세: GLOSSARY.md)
