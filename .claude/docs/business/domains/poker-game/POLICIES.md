# 포커 게임 도메인 - 서비스 정책서

## poker.방_유효성_검사

### Validation 규칙

#### POL-001: 신규 접근자는 서버에서 방 존재 여부 확인
- **규칙**: 초대 링크로 방에 접근하는 사용자는 반드시 서버에 방 존재 여부를 확인한다. 서버 메모리에 해당 방이 없으면 유효하지 않은 방으로 처리한다.
- **위반 시**: "방을 찾을 수 없습니다 / 존재하지 않거나 이미 종료된 방입니다." 화면이 표시되고 게임 UI에 접근할 수 없다.
- **비즈니스 배경**: 서버 재시작 또는 마지막 참가자 이탈 후 방이 자동 정리되므로, 잘못된 링크로 진입한 사용자가 빈 게임 화면을 보는 상황을 방지한다.

#### POL-002: 이미 입장한 사용자는 서버 재확인 생략
- **규칙**: 브라우저 세션에 이름과 방 ID가 이미 저장된 사용자는 서버 재확인 없이 즉시 방으로 진입한다.
- **비즈니스 배경**: 새로고침한 기존 참가자가 불필요하게 입장 폼으로 이동하는 것을 방지한다.

#### POL-003: 상태 복원 완료 전 방 검사 수행 금지
- **규칙**: 브라우저 세션 스토리지에서 게임 상태 복원이 완료되기 전까지는 방 유효성 검사를 시작하지 않는다. 복원 중에는 로딩 화면을 표시한다.
- **비즈니스 배경**: 서버 사이드 렌더링과 클라이언트 상태 복원의 타이밍 차이로 인한 화면 불일치를 방지한다.

## poker.방_입장_참가자

### Validation 규칙

#### POL-004: 이름 공백 입력 차단
- **규칙**: 이름 필드에 공백만 입력하거나 비어 있으면 참여하기 버튼이 비활성화되어 입장할 수 없다.
- **위반 시**: 참여하기 버튼이 비활성 상태로 유지되며 입장이 차단된다.
- **비즈니스 배경**: 이름이 없는 참가자가 방에 들어오면 다른 참가자가 구별할 수 없으므로 최소한의 식별 정보를 요구한다.

#### POL-005: 이름 앞뒤 공백 자동 제거
- **규칙**: 입력된 이름은 앞뒤 공백을 제거한 후 저장된다.
- **비즈니스 배경**: 실수로 추가된 공백이 호스트 재접속 이름 매칭이나 표시에 영향을 주지 않도록 한다.

#### POL-006: 최근 입력 이름 자동 캐싱
- **규칙**: 참가자가 이름을 입력하고 입장하면 브라우저에 이름이 저장된다. 다음 방 링크 접근 시 저장된 이름이 자동으로 입력 필드에 채워진다.
- **위반 시**: 프라이빗 모드 또는 브라우저 저장소 비활성 환경에서는 자동 입력 없이 빈 칸으로 표시된다.
- **비즈니스 배경**: 반복적으로 세션에 참가하는 사용자가 매번 이름을 입력하는 불편을 줄인다.

#### POL-007: 동일 참가자 중복 등록 방지
- **규칙**: 이미 참가자 목록에 존재하는 ID를 가진 사용자는 중복으로 추가되지 않는다.
- **비즈니스 배경**: SSE 이벤트와 WebRTC 연결 콜백이 거의 동시에 도착하는 경우 같은 참가자가 목록에 두 번 나타나는 것을 방지한다.

## poker.방_대기_화면

### 상태 정책

#### POL-008: 2인 미만 시 게임 진행 차단
- **규칙**: 방에 참가자가 2명 미만이면 포커 테이블과 카드덱을 표시하지 않고 대기 화면을 표시한다.
- **전이**: 대기 화면 → 게임 진행
- **전이 조건**: 총 참가자 수가 2명 이상이 되는 시점에 자동 전환
- **역전이**: 가능 — 참가자 이탈로 인원이 1명 이하로 감소하면 대기 화면으로 복귀
- **영향**: 2명 이상이 되는 순간 포커 테이블, 카드덱, 티켓 패널이 렌더링된다.
- **비즈니스 배경**: Planning Poker는 팀 합의가 목적이므로 최소 2인이 참여한 투표에서만 의미 있는 결과가 도출된다.

#### POL-009: 호스트와 참가자에게 다른 대기 화면 표시
- **규칙**: 대기 화면에서 호스트에게는 초대 링크 복사 UI를, 일반 참가자에게는 "호스트와 연결 중" 안내를 표시한다.

## poker.카드_선택

### 설정값

#### POL-010: 카드 덱 구성 — Fibonacci 수열 + 특수값 9장 고정
- **규칙**: 투표에 사용할 수 있는 카드는 1, 2, 3, 5, 8, 13, 21과 특수값 '?'(추정 불가), '☕'(휴식 필요) 총 9장으로 고정된다.
- **비즈니스 의미**: Fibonacci 수열 기반 상대적 크기 추정으로 큰 작업의 불확실성을 반영하고, 특수값은 추정이 불가하거나 논의가 더 필요한 상황을 표현한다.
- **변경 시 영향**: 카드 종류 변경 시 Average 계산 로직(비숫자 값 제외)도 함께 검토해야 한다.

### Validation 규칙

#### POL-011: 투표 공개 이후 카드 선택 불가
- **규칙**: 투표 결과가 공개되면 모든 카드가 비활성화되어 추가 선택이 불가능하다.
- **위반 시**: 해당 없음 — UI에서 클릭 이벤트가 차단된다.
- **비즈니스 배경**: 결과 공개 후 투표를 변경하는 것을 방지하여 결과의 신뢰성을 보장한다.

## poker.투표_공개

### 상태 정책

#### POL-012: 전원 투표 완료 및 2인 이상 조건 충족 시 자동 공개 시작
- **규칙**: 투표 단계에서 참가자가 2명 이상이고 모든 참가자가 카드를 선택하면 자동 공개 프로세스가 시작된다.
- **전이**: 투표 중(voting) → 결과 공개됨(revealed)
- **전이 조건**: 전원 투표 완료 감지 후 2초 카운트다운 종료
- **역전이**: 가능 — 재투표(Re-vote) 실행 시 투표 중으로 복귀
- **영향**: 카운트다운 표시가 포커 테이블 중앙에 표시되고, 0이 되면 모든 참가자의 투표값이 공개된다.

#### POL-013: 카운트다운 중 조건 미충족 시 취소
- **규칙**: 카운트다운 진행 중 참가자가 이탈하여 전원 투표 완료 조건이 더 이상 충족되지 않으면 카운트다운이 즉시 취소되고 투표 대기 상태로 돌아간다.
- **비즈니스 배경**: 이탈한 참가자의 투표가 포함된 채로 결과가 공개되는 것을 방지한다.

### 설정값

#### POL-014: 전원 투표 완료 후 2초 카운트다운
- **규칙**: 전원 투표 완료 감지 후 2초 카운트다운이 포커 테이블 중앙에 표시된다. 카운트다운이 0에 도달하면 투표가 공개된다.
- **비즈니스 의미**: 동시에 투표가 완료되는 시점을 모든 참가자가 인지할 수 있도록 시각적 신호를 제공한다.
- **변경 시 영향**: 카운트다운 시간을 줄이면 결과가 더 빠르게 공개되고, 늘리면 참가자가 결과를 확인할 준비를 할 수 있는 시간이 늘어난다.

### Validation 규칙

#### POL-015: 투표값 없을 시 '?'로 공개
- **규칙**: 공개 시점에 투표값이 없는 참가자의 투표는 '?'로 처리되어 전송된다.

## poker.결과_표시

### 설정값

#### POL-016: Mode 계산 — 최초 최빈값 우선
- **규칙**: Mode는 가장 많이 선택된 투표값이다. 동점인 경우 참가자 목록 순서상 먼저 등장한 값이 Mode로 선택된다.
- **비즈니스 의미**: Mode는 팀의 공통 의견을 대표하며 스토리 포인트 결정의 기준값으로 활용된다.

#### POL-017: Average 계산 — 숫자 투표만 포함
- **규칙**: Average는 숫자로 변환 가능한 투표값(1, 2, 3, 5, 8, 13, 21)만 대상으로 계산된다. '?'나 '☕' 같은 비숫자 값은 평균 계산에서 제외된다. 숫자 투표가 하나도 없으면 Average는 표시되지 않는다.
- **비즈니스 배경**: 의사 결정이 불가한 특수 투표값을 평균에서 제외하여 의미 있는 수치만 집계한다.

### Validation 규칙

#### POL-018: 결과 화면 컨트롤은 호스트 전용
- **규칙**: 투표 결과 공개 후 Re-vote(재투표)와 Next(다음 티켓) 버튼은 호스트에게만 표시된다. 일반 참가자에게는 "호스트만 진행 가능" 안내 문구가 표시된다.
- **위반 시**: 해당 없음 — 버튼 자체가 표시되지 않는다.
- **비즈니스 배경**: 라운드 진행 결정권을 호스트에게 집중시켜 게임 흐름의 일관성을 유지한다.

#### POL-019: 마지막 티켓에서는 Next 버튼 미표시
- **규칙**: 현재 티켓이 마지막 티켓이면 Next 버튼 대신 "All Done" 텍스트가 표시된다. 이 텍스트는 버튼이 아니므로 클릭해도 동작하지 않는다.
- **비즈니스 배경**: 더 이상 진행할 티켓이 없음을 명확히 알리고, 실수로 Next를 클릭하여 의도치 않게 세션이 종료되는 것을 방지한다.

## poker.재투표

### 상태 정책

#### POL-020: 재투표 시 현재 라운드만 초기화, 완료 이력 보존
- **규칙**: 재투표가 실행되면 모든 참가자의 투표값과 투표 완료 여부가 초기화되고, 단계가 투표 중으로 돌아간다. 이미 완료된 이전 티켓의 결과는 영향을 받지 않으며, 현재 티켓은 완료 목록에 기록되지 않는다.
- **전이**: 결과 공개됨(revealed) → 투표 중(voting)
- **전이 조건**: 호스트가 Re-vote 버튼 클릭
- **역전이**: 불가 — 초기화 후 이전 투표값을 복구할 수 없다.
- **영향**: 모든 참가자의 카드 선택이 취소되고 카드덱이 다시 활성화된다.

### Validation 규칙

#### POL-021: 재투표는 호스트만 실행 가능
- **규칙**: 재투표는 투표 결과가 공개된 상태에서 호스트만 실행할 수 있다.
- **위반 시**: 해당 없음 — 일반 참가자에게는 버튼이 표시되지 않는다.

## poker.다음_티켓

### 상태 정책

#### POL-022: 다음 티켓 이동 시 현재 결과를 불변 스냅샷으로 기록
- **규칙**: 다음 티켓으로 이동할 때 현재 티켓의 투표 결과(참가자별 투표값, Mode, Average)가 완료 목록에 스냅샷으로 기록된다. 기록 후에는 변경되지 않는다.
- **전이**: 현재 티켓 결과 공개 → 다음 티켓 투표 중
- **전이 조건**: 호스트가 Next 버튼 클릭 (마지막 티켓이 아닐 때)
- **역전이**: 불가 — 이전 티켓으로 되돌아갈 수 없다.
- **영향**: 완료된 티켓이 티켓 패널의 히스토리에 추가되고 새 티켓에 대한 투표가 시작된다.

#### POL-023: 결과 계산 순서 보장 — 스냅샷 먼저, 초기화 나중
- **규칙**: 다음 티켓 이동 시 Mode와 Average는 참가자 투표 상태가 초기화되기 전에 계산된다. Mode가 없으면 '?', Average가 없으면 0으로 스냅샷에 저장된다.
- **비즈니스 배경**: 투표 초기화 이후에 계산하면 빈 결과가 저장되는 오류가 발생한다.

### Validation 규칙

#### POL-024: 다음 티켓 이동은 호스트 전용, 마지막 티켓 제외
- **규칙**: 다음 티켓으로 이동하는 권한은 호스트에게만 있으며, 투표 결과가 공개된 상태여야 한다. 마지막 티켓에서는 Next 버튼이 표시되지 않는다.
- **위반 시**: 해당 없음 — 조건 미충족 시 버튼이 표시되지 않는다.

## poker.세션_완료

### 상태 정책

#### POL-025: 마지막 티켓 완료 후 세션 완료 화면 전환
- **규칙**: 마지막 티켓에 대해 Next를 실행하면 더 이상 진행할 티켓이 없어 세션 완료 화면으로 전환된다. 티켓이 한 장도 없는 경우는 세션 완료로 간주하지 않는다.
- **전이**: 게임 진행 → 세션 완료
- **전이 조건**: 마지막 티켓의 Next 실행 후 현재 티켓이 없어진 시점
- **역전이**: 불가 — 세션 종료 후 홈 화면으로만 이동한다.
- **영향**: 완료된 티켓 목록(Mode, Average 포함)과 총 스토리 포인트 합계가 표시된다.

### 설정값

#### POL-026: 총 스토리 포인트 계산 — 비숫자 Mode는 0으로 처리
- **규칙**: 세션 완료 화면의 총 스토리 포인트 합계 계산 시, Mode가 숫자가 아닌 경우 해당 티켓의 포인트는 0으로 처리하여 합산한다.
- **비즈니스 의미**: 스토리 포인트를 확정하지 못한 티켓을 합계에서 제외하되 총합 계산 자체는 유지한다.

### Validation 규칙

#### POL-027: 세션 종료 시 로컬 상태 전체 초기화
- **규칙**: 세션 종료 버튼을 클릭하면 게임 상태 전체가 초기값으로 리셋되고 브라우저 세션 스토리지에서 방 데이터가 삭제된 후 홈 화면으로 이동한다.
- **비즈니스 의미**: 이전 세션의 데이터가 남아 다음 세션에 간섭하지 않도록 완전히 정리한다.

## poker.티켓_패널

### 상태 정책

#### POL-028: 티켓이 없으면 패널 미렌더링

- **비즈니스 배경**: 표시할 내용이 없는 패널을 렌더링해 사용자를 혼란스럽게 하지 않는다.

## poker.참가자_추방

### Validation 규칙

#### POL-029: 추방은 호스트만 가능, 자기 자신 제외
- **규칙**: 추방 버튼은 호스트에게만 표시되며, 호스트 자신의 좌석에는 표시되지 않는다.
- **위반 시**: 해당 없음 — 버튼 자체가 표시되지 않는다.
- **비즈니스 배경**: 게임 진행 통제권을 호스트에게 부여하되, 실수로 자기 자신을 추방하는 상황을 방지한다.

### 에러 시나리오

#### ERR-001: 방에서 추방됨
- **메시지**: "방에서 추방되었습니다 / 호스트에 의해 추방되었습니다."
- **발생 상황**: 호스트가 해당 참가자의 추방 버튼을 클릭했을 때
- **사용자 영향**: 전체 화면 오버레이로 추방 메시지가 표시되고 게임 진행이 불가능해진다.
- **대응 방법**: "홈으로 돌아가기" 버튼으로 홈 화면에서 새 방에 참가할 수 있다.

## poker.방_종료_및_이탈

### Validation 규칙

#### POL-030: 호스트 방 종료 전 확인 다이얼로그 표시
- **규칙**: 호스트가 방 종료 버튼을 누르면 "방을 종료하시겠습니까? 모든 참가자의 연결이 끊어집니다." 확인 다이얼로그가 표시된다. 취소를 선택하면 방은 유지된다.
- **위반 시**: 취소 선택 시 방 유지, 확인 선택 시 즉시 방 종료 진행
- **비즈니스 배경**: 실수로 방 종료 버튼을 클릭하여 진행 중인 세션이 종료되는 것을 방지한다.

### 상태 정책

#### POL-031: 호스트 방 종료 시 모든 참가자 연결 종료
- **규칙**: 호스트가 방 종료를 확인하면 모든 참가자에게 방 종료 알림이 전달된다.
- **전이**: 게임 진행 중 → 방 종료 화면
- **전이 조건**: 호스트가 방 종료를 확인할 때
- **역전이**: 불가 — 종료된 방은 복구할 수 없다.
- **영향**: 모든 참가자 화면에 "방이 종료되었습니다" 오버레이가 표시되고 게임 진행이 차단된다.

#### POL-032: 탭 닫힘 시 서버와 P2P 양쪽으로 이탈 알림 전송
- **규칙**: 사용자가 탭을 닫거나 브라우저를 종료할 때 서버 API와 P2P DataChannel 두 경로로 이탈을 알린다. 이중 알림으로 인한 중복 처리는 서버 측 중복 방지 로직으로 처리된다.
- **비즈니스 배경**: 브라우저가 닫힐 때 비동기 API 호출은 취소될 수 있으므로, 신뢰성 있는 이탈 알림을 보장한다.

### 에러 시나리오

#### ERR-002: 방이 종료됨
- **메시지**: "방이 종료되었습니다 / 호스트가 방을 나갔습니다."
- **발생 상황**: 호스트가 방 종료 버튼을 눌러 게임을 종료했을 때
- **사용자 영향**: 전체 화면 오버레이로 종료 메시지가 표시되고 게임을 계속할 수 없다.
- **대응 방법**: "홈으로 돌아가기" 버튼으로 홈 화면에서 새 방에 참가할 수 있다.

## poker.호스트_재접속

### 상태 정책

#### POL-033: 호스트 이탈 시 참가자에게 무기한 대기 오버레이 표시
- **규칙**: 호스트가 연결을 끊으면 남아 있는 참가자에게 "호스트 재접속 대기 중" 오버레이가 표시된다. 오버레이는 호스트가 재연결할 때까지 무기한 지속되며, 참가자는 "홈으로 돌아가기" 버튼으로 스스로 이탈할 수 있다.
- **전이**: 게임 진행 → 호스트 대기 오버레이
- **전이 조건**: 호스트 피어의 연결이 끊길 때 자동 전환
- **역전이**: 가능 — 호스트 재접속 시 대기 오버레이가 해제되고 게임이 재개된다.
- **영향**: 게임이 일시 중단되고 카드 선택, 투표 등 모든 게임 동작이 오버레이로 가려진다.

#### POL-034: 이름 완전 일치로 호스트 권한 자동 복원
- **규칙**: 호스트 대기 중 새로운 참가자가 연결될 때, 그 이름이 이탈한 호스트의 이름과 대소문자를 포함한 완전 일치하면 자동으로 호스트 권한이 복원된다. 복원 후 모든 참가자에게 호스트 변경이 전파된다.
- **전이**: 호스트 대기 → 게임 진행
- **전이 조건**: 이탈한 호스트와 동일한 이름으로 새 피어 연결
- **역전이**: 해당 없음
- **영향**: 호스트 대기 오버레이가 사라지고 게임 진행 컨트롤(Re-vote, Next, 추방)이 복원된다.
- **비즈니스 배경**: 이름 기반 매칭이므로 동일한 이름을 가진 다른 사람이 먼저 입장하면 의도치 않게 호스트 권한이 부여될 수 있는 한계가 있다.

### 설정값

#### POL-035: 호스트 복원 브로드캐스트 500ms 지연 전송
- **규칙**: 호스트 재접속이 확인되면 즉시가 아닌 500ms 후에 호스트 변경 알림이 전체 참가자에게 전달된다.
- **비즈니스 배경**: P2P DataChannel이 완전히 열리기 전에 메시지를 전송하면 메시지가 유실될 수 있어, DataChannel 준비 완료를 기다리는 안전 마진을 둔다.

## poker.p2p_연결_및_릴레이

### 상태 정책

#### POL-036: WebRTC P2P 실패 시 서버 릴레이 자동 전환
- **규칙**: WebRTC 연결 시도 후 8초 이내에 직접 연결이 열리지 않으면 자동으로 서버 릴레이 모드로 전환된다. 릴레이 모드 전환 후에는 세션 내에서 P2P로 복귀하지 않는다.
- **전이**: P2P 연결 시도(connecting) → 서버 릴레이(relay)
- **전이 조건**: P2P 연결 8초 타임아웃
- **역전이**: 불가 — 릴레이 모드로 전환 후에는 복귀하지 않는다.
- **영향**: 사용자에게는 게임 기능상 차이가 없으나 서버를 통해 메시지가 전달된다.
- **비즈니스 배경**: 방화벽이나 NAT 환경에서 P2P 연결이 불가능한 사용자도 게임에 참여할 수 있도록 보장한다.

### 설정값

#### POL-037: P2P 연결 폴백 대기 시간 — 8초
- **규칙**: WebRTC P2P 연결 시도 후 8초 동안 DataChannel이 열리지 않으면 서버 릴레이로 전환한다.
- **비즈니스 의미**: 네트워크 환경에 따라 P2P 연결이 실패할 때 최대 8초 후 게임 메시지 전달이 보장된다.
- **변경 시 영향**: 줄이면 P2P 연결 성공 가능성이 있는 환경에서도 릴레이로 전환될 수 있고, 늘리면 연결 장애 시 사용자가 더 오래 대기하게 된다.

### Validation 규칙

#### POL-038: 신규 참가자만 상태 동기화 요청 전송
- **규칙**: 방에 새로 입장한 참가자만 기존 참가자에게 상태 동기화를 요청한다. 기존 참가자는 동기화 요청을 보내지 않는다.
- **비즈니스 배경**: 기존 참가자가 동기화 요청을 보내면 빈 상태 응답이 돌아와 신규 참가자의 게임 상태가 초기화될 수 있다.

#### POL-039: 동기화 수신 시 자신의 투표 상태 유지
- **규칙**: 방 상태 동기화 메시지를 수신할 때 자신의 투표 상태(로컬에 저장된 상태)는 동기화 데이터에 의해 덮어씌워지지 않고 유지된다.
- **비즈니스 배경**: 동기화 중 자신의 투표값이 유실되는 것을 방지한다.

## 에러 시나리오 (전체)

#### ERR-003: 방을 찾을 수 없음
- **메시지**: "방을 찾을 수 없습니다 / 존재하지 않거나 이미 종료된 방입니다."
- **발생 상황**: 초대 링크로 접근했으나 서버에 해당 방이 존재하지 않을 때. 방이 생성된 적 없거나 모든 참가자가 나가서 방이 자동 삭제된 경우.
- **사용자 영향**: 방 없음 전용 화면이 표시되고 게임에 진입할 수 없다.
- **대응 방법**: "홈으로 돌아가기" 버튼을 클릭하여 새 방을 생성하거나 유효한 초대 링크를 받는다.
