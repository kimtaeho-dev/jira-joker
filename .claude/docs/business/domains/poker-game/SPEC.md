# 포커 게임 도메인 - 기획서

> 최종 갱신: 2026-03-29

## 도메인 개요

Planning Poker 세션의 핵심 게임 흐름을 담당한다. 방 생성/참가, 카드 투표, 결과 공개, 티켓 진행, 세션 완료까지 전체 라이프사이클을 관리하며, 실시간 통신을 통해 참가자 간 상태를 동기화한다.

---

## 방 유효성 검사

### 이 기능은 무엇인가
초대 링크로 방에 접근할 때 해당 방이 실제로 존재하는지 확인하고, 결과에 따라 입장 폼 또는 안내 화면을 표시하는 기능이다.

### 사용자 시나리오
1. 사용자가 초대 링크를 통해 방 페이지에 접근한다
2. 브라우저에 이전 세션 정보가 있으면 재확인 없이 바로 진입한다
3. 새로운 접근이면 서버에 방 존재 여부를 확인한다
4. 방이 있으면 이름 입력 폼이, 없으면 "방을 찾을 수 없습니다" 화면이 표시된다

### 왜 이렇게 동작하는가
- **기존 사용자 즉시 진입**: 새로고침한 사용자가 불필요하게 입장 폼을 다시 거치지 않도록 한다
- **상태 복원 대기**: 브라우저 세션 스토리지 복원이 완료되기 전 검사를 시작하면 잘못된 판단이 발생할 수 있다

### 주요 정책
- POL-001: 신규 접근자는 서버에서 방 존재 여부 확인 (상세: POLICIES.md)
- POL-002: 이미 입장한 사용자는 서버 재확인 생략 (상세: POLICIES.md)
- POL-003: 상태 복원 완료 전 방 검사 수행 금지 (상세: POLICIES.md)

### 에러 상황
- ERR-003: 방을 찾을 수 없음 — 홈으로 돌아가 새 방을 생성하거나 유효한 링크 요청

### 관련 용어
- 방: Planning Poker 세션이 진행되는 가상 공간 (상세: GLOSSARY.md)
- 세션 스토리지: 브라우저 탭 단위 게임 상태 저장 공간 (상세: GLOSSARY.md)

## 방 입장 (참가자)

### 이 기능은 무엇인가
초대 링크로 방에 접근한 신규 참가자가 이름을 입력하고 방에 참가하는 기능이다. 최근 입력한 이름은 자동으로 캐싱된다.

### 사용자 시나리오
1. 방이 유효하면 이름 입력 폼이 표시된다
2. 이전에 입력한 이름이 있으면 자동으로 채워진다
3. 이름을 입력하고 "참여하기" 버튼을 클릭한다
4. 방에 입장하고 실시간 연결이 시작된다

### 왜 이렇게 동작하는가
- **이름 필수**: 참가자를 구별할 수 있는 최소 식별 정보이자, 호스트 재접속 시 이름 매칭의 기반이다
- **공백 자동 제거**: 실수로 추가된 공백이 이름 매칭에 영향을 주지 않도록 한다
- **중복 등록 방지**: SSE와 WebRTC 콜백이 동시에 도착할 때 같은 참가자가 두 번 표시되지 않도록 한다

### 주요 정책
- POL-004: 이름 공백 입력 차단 (상세: POLICIES.md)
- POL-005: 이름 앞뒤 공백 자동 제거 (상세: POLICIES.md)
- POL-006: 최근 입력 이름 자동 캐싱 (상세: POLICIES.md)
- POL-007: 동일 참가자 중복 등록 방지 (상세: POLICIES.md)

### 관련 용어
- 참가자: Planning Poker 방에 입장한 모든 사용자 (상세: GLOSSARY.md)

## 방 대기 화면

### 이 기능은 무엇인가
참가자가 2명 미만일 때 게임을 시작하지 않고 대기 화면을 표시하는 기능이다. 호스트에게는 초대 링크를, 참가자에게는 연결 대기 안내를 보여준다.

### 사용자 시나리오
1. 호스트가 방을 만들면 대기 화면이 표시된다
2. 호스트에게 초대 링크 복사 UI가 표시된다
3. 참가자가 들어오면 연결 대기 안내가 표시된다
4. 총 인원이 2명 이상이 되면 자동으로 게임 화면으로 전환된다

### 왜 이렇게 동작하는가
- **최소 2인**: Planning Poker는 팀 합의가 목적이므로 혼자서는 의미 있는 추정이 불가능하다

### 주요 정책
- POL-008: 2인 미만 시 게임 진행 차단 (상세: POLICIES.md)
- POL-009: 호스트와 참가자에게 다른 대기 화면 표시 (상세: POLICIES.md)

### 관련 용어
- 대기 화면: 인원 부족 시 표시되는 화면 (상세: GLOSSARY.md)
- 호스트: 방을 생성한 참가자 (상세: GLOSSARY.md)

## 카드 선택 (투표)

### 이 기능은 무엇인가
참가자가 Fibonacci 수열 + 특수값으로 구성된 카드 덱에서 추정값을 선택하는 기능이다. 선택 즉시 다른 참가자에게 투표 완료가 알려지지만, 실제 값은 공개 전까지 숨겨진다.

### 사용자 시나리오
1. 화면 하단에 9장의 카드(1, 2, 3, 5, 8, 13, 21, ?, ☕)가 표시된다
2. 카드를 클릭하면 선택이 완료되고 다른 참가자에게 알려진다
3. 다른 카드를 클릭하면 변경할 수 있다
4. 모든 참가자의 좌석에 투표 완료 여부가 표시된다

### 왜 이렇게 동작하는가
- **Fibonacci 수열**: 숫자 간 격차가 커질수록 불확실성이 반영되어 상대적 크기 추정에 적합하다
- **특수값 '?', '☕'**: 추정이 불가능하거나 논의가 더 필요한 상황을 표현한다
- **공개 전 비공개**: 다른 참가자의 투표에 영향을 받지 않는 독립적 추정을 보장한다

### 주요 정책
- POL-010: 카드 덱 구성 Fibonacci + 특수값 9장 고정 (상세: POLICIES.md)
- POL-011: 투표 공개 이후 카드 선택 불가 (상세: POLICIES.md)

### 관련 용어
- 카드 덱: 투표에 사용할 수 있는 카드 모음 (상세: GLOSSARY.md)
- Fibonacci 수열: 카드 덱의 숫자 기반 (상세: GLOSSARY.md)

## 투표 공개

### 이 기능은 무엇인가
모든 참가자가 카드를 선택하면 2초 카운트다운 후 자동으로 모든 투표값을 동시에 공개하는 기능이다.

### 사용자 시나리오
1. 마지막 참가자가 카드를 선택한다
2. 포커 테이블 중앙에 2초 카운트다운이 표시된다
3. 0에 도달하면 모든 참가자의 투표값이 동시에 공개된다
4. Mode(최빈값)와 Average(평균)가 테이블 중앙에 표시된다

### 왜 이렇게 동작하는가
- **자동 공개**: 호스트가 별도로 공개 버튼을 누르지 않아도 전원 투표 완료 시 자동 진행된다
- **2초 카운트다운**: 참가자가 공개 시점을 인지하고 준비할 수 있는 시간을 제공한다
- **카운트다운 취소**: 진행 중 참가자가 이탈하면 조건 미충족으로 취소되어 불완전한 결과 공개를 방지한다

### 주요 정책
- POL-012: 전원 투표 완료 및 2인 이상 시 자동 공개 시작 (상세: POLICIES.md)
- POL-013: 카운트다운 중 조건 미충족 시 취소 (상세: POLICIES.md)
- POL-014: 전원 투표 완료 후 2초 카운트다운 (상세: POLICIES.md)
- POL-015: 투표값 없을 시 '?'로 공개 (상세: POLICIES.md)

### 관련 용어
- 결과 공개: 투표값을 동시에 보이게 전환하는 동작 (상세: GLOSSARY.md)
- 투표 단계: 투표 중 / 공개됨 두 가지 상태 (상세: GLOSSARY.md)

## 결과 표시

### 이 기능은 무엇인가
투표가 공개되면 Mode(최빈값)와 Average(숫자 평균)를 계산하여 표시하고, 호스트에게 다음 행동(재투표 또는 다음 티켓)을 선택할 수 있는 컨트롤을 제공하는 기능이다.

### 사용자 시나리오
1. 투표가 공개되면 각 참가자 좌석에 선택한 값이 표시된다
2. 테이블 중앙에 Mode와 Average가 표시된다
3. 호스트에게 Re-vote와 Next 버튼이 표시된다
4. 일반 참가자에게는 "호스트만 진행 가능" 안내가 표시된다
5. 마지막 티켓에서는 Next 대신 "All Done" 텍스트가 표시된다

### 왜 이렇게 동작하는가
- **호스트 전용 컨트롤**: 라운드 진행 결정권을 호스트에게 집중시켜 게임 흐름의 일관성을 유지한다
- **Mode 우선**: 팀의 공통 의견을 대표하는 값으로 스토리 포인트 결정 기준에 적합하다
- **숫자만 Average**: '?'와 '☕'를 평균에서 제외하여 의미 있는 수치만 집계한다

### 주요 정책
- POL-016: Mode 계산 — 동점 시 최초 최빈값 우선 (상세: POLICIES.md)
- POL-017: Average 계산 — 숫자 투표만 포함 (상세: POLICIES.md)
- POL-018: 결과 화면 컨트롤은 호스트 전용 (상세: POLICIES.md)
- POL-019: 마지막 티켓에서는 Next 버튼 미표시 (상세: POLICIES.md)

### 관련 용어
- 모드: 가장 많이 선택된 투표값 (상세: GLOSSARY.md)
- 평균: 숫자 투표값의 산술 평균 (상세: GLOSSARY.md)

## 재투표

### 이 기능은 무엇인가
호스트가 현재 티켓에 대한 투표를 초기화하여 다시 투표할 수 있도록 하는 기능이다. 이전 티켓의 결과에는 영향을 주지 않는다.

### 사용자 시나리오
1. 결과 공개 후 팀이 충분한 토론이 필요하다고 판단한다
2. 호스트가 Re-vote 버튼을 클릭한다
3. 모든 참가자의 투표가 초기화되고 카드덱이 다시 활성화된다
4. 참가자들이 다시 카드를 선택한다

### 왜 이렇게 동작하는가
- **현재 라운드만 초기화**: 재투표는 현재 티켓의 투표만 초기화하며, 이미 완료된 이전 티켓의 결과는 보존된다
- **완료 미기록**: 재투표로 이동한 라운드의 이전 결과는 히스토리에 기록되지 않는다

### 주요 정책
- POL-020: 재투표 시 현재 라운드만 초기화, 완료 이력 보존 (상세: POLICIES.md)
- POL-021: 재투표는 호스트만 실행 가능 (상세: POLICIES.md)

### 관련 용어
- 재투표: 현재 티켓 투표를 초기화하는 동작 (상세: GLOSSARY.md)

## 다음 티켓 이동

### 이 기능은 무엇인가
현재 티켓의 투표 결과를 스냅샷으로 기록하고 다음 티켓으로 진행하는 기능이다. 결과 기록 후 투표 상태가 초기화되어 새로운 라운드가 시작된다.

### 사용자 시나리오
1. 결과 공개 후 호스트가 Next 버튼을 클릭한다
2. 현재 티켓의 투표 결과(참가자별 투표, Mode, Average)가 히스토리에 기록된다
3. 다음 티켓이 표시되고 투표 상태가 초기화된다
4. 참가자들이 새 티켓에 대해 카드를 선택한다

### 왜 이렇게 동작하는가
- **불변 스냅샷**: 완료된 티켓의 결과는 한 번 기록되면 수정되지 않는다
- **계산 순서 보장**: Mode와 Average를 참가자 상태 초기화 전에 계산하여 빈 결과가 저장되지 않도록 한다

### 주요 정책
- POL-022: 다음 티켓 이동 시 현재 결과를 불변 스냅샷으로 기록 (상세: POLICIES.md)
- POL-023: 결과 계산 순서 보장 (상세: POLICIES.md)
- POL-024: 다음 티켓 이동은 호스트 전용 (상세: POLICIES.md)

### 관련 용어
- 완료 티켓: 투표 완료 후 생성되는 불변 결과 스냅샷 (상세: GLOSSARY.md)

## 세션 완료

### 이 기능은 무엇인가
모든 티켓의 추정이 완료되면 세션 요약 화면을 표시하는 기능이다. 각 티켓의 Mode, Average와 총 스토리 포인트 합계를 보여준다.

### 사용자 시나리오
1. 마지막 티켓에 대해 Next를 실행한다
2. 세션 완료 화면이 표시된다
3. 완료된 티켓 목록과 각 결과(Mode, Average)가 테이블로 표시된다
4. 총 스토리 포인트 합계가 표시된다
5. "세션 종료" 버튼을 클릭하면 게임 상태가 초기화되고 홈으로 이동한다

### 왜 이렇게 동작하는가
- **비숫자 Mode는 0**: '?' 등으로 결정된 티켓은 합계에서 제외하되 총합 계산 자체는 유지한다
- **완전 초기화**: 세션 종료 시 이전 세션 데이터가 남아 다음 세션에 간섭하지 않도록 정리한다

### 주요 정책
- POL-025: 마지막 티켓 완료 후 세션 완료 화면 전환 (상세: POLICIES.md)
- POL-026: 총 스토리 포인트 — 비숫자 Mode는 0 처리 (상세: POLICIES.md)
- POL-027: 세션 종료 시 로컬 상태 전체 초기화 (상세: POLICIES.md)

### 관련 용어
- 세션: 전체 Planning Poker 진행 과정 (상세: GLOSSARY.md)
- 스토리 포인트: 이슈의 상대적 작업량 추정 단위 (상세: GLOSSARY.md)

## 티켓 패널

### 이 기능은 무엇인가
화면 우측에 현재 진행 중인 티켓의 상세 정보와 이전 완료 티켓 히스토리를 표시하는 패널이다. 토글 버튼으로 열고 닫을 수 있다.

### 사용자 시나리오
1. 게임 진행 중 우측 토글 버튼을 클릭하면 패널이 열린다
2. 현재 티켓의 키, 제목, 설명, 담당자, 우선순위가 표시된다
3. 아래에 완료된 티켓의 결과 히스토리가 표시된다
4. 모바일에서는 오버레이로 표시되며 배경 클릭 시 닫힌다

### 왜 이렇게 동작하는가
- **티켓 없으면 미표시**: 표시할 내용이 없으면 패널 자체를 렌더링하지 않는다

### 주요 정책
- POL-028: 티켓이 없으면 패널 미렌더링 (상세: POLICIES.md)

## 참가자 추방

### 이 기능은 무엇인가
호스트가 응답이 없거나 문제가 있는 참가자를 방에서 강제로 내보내는 기능이다.

### 사용자 시나리오
1. 포커 테이블에서 참가자 좌석에 마우스를 올린다
2. 호스트에게만 추방 버튼이 표시된다 (자기 자신 제외)
3. 추방 버튼을 클릭하면 해당 참가자에게 추방 메시지가 전달된다
4. 추방된 참가자에게 "방에서 추방되었습니다" 화면이 표시된다

### 왜 이렇게 동작하는가
- **호스트 전용**: 게임 진행 통제권을 호스트에게 부여한다
- **자기 추방 방지**: 실수로 자기 자신을 추방하는 상황을 방지한다

### 주요 정책
- POL-029: 추방은 호스트만 가능, 자기 자신 제외 (상세: POLICIES.md)

### 에러 상황
- ERR-001: 방에서 추방됨 — "홈으로 돌아가기" 버튼으로 새 방에 참가 가능

### 관련 용어
- 추방: 호스트가 참가자를 강제 퇴장시키는 동작 (상세: GLOSSARY.md)

## 방 종료 및 이탈

### 이 기능은 무엇인가
호스트의 방 종료와 일반 참가자의 이탈을 처리하는 기능이다. 호스트가 방을 종료하면 모든 참가자의 연결이 끊기고, 개별 참가자의 이탈은 나머지 참가자에게 알린다.

### 사용자 시나리오
1. 호스트가 "나가기" 버튼을 클릭하면 확인 다이얼로그가 표시된다
2. 확인 시 모든 참가자에게 방 종료 알림이 전달된다
3. 일반 참가자가 "나가기" 버튼을 클릭하면 즉시 이탈한다
4. 브라우저를 직접 닫아도 서버와 P2P 양쪽으로 이탈이 알려진다

### 왜 이렇게 동작하는가
- **호스트 확인 다이얼로그**: 실수로 세션을 종료하는 것을 방지한다
- **이중 경로 이탈 알림**: 비동기 API 호출이 취소될 수 있는 브라우저 종료 시나리오에서도 이탈 감지를 보장한다

### 주요 정책
- POL-030: 호스트 방 종료 전 확인 다이얼로그 (상세: POLICIES.md)
- POL-031: 호스트 방 종료 시 모든 참가자 연결 종료 (상세: POLICIES.md)
- POL-032: 탭 닫힘 시 서버와 P2P 양쪽으로 이탈 알림 (상세: POLICIES.md)

### 에러 상황
- ERR-002: 방이 종료됨 — "홈으로 돌아가기" 버튼으로 새 방에 참가 가능

## 호스트 재접속

### 이 기능은 무엇인가
호스트가 이탈한 후 같은 이름으로 재접속하면 자동으로 호스트 권한이 복원되는 기능이다. 호스트 부재 중 참가자에게는 대기 오버레이가 표시된다.

### 사용자 시나리오
1. 호스트가 실수로 탭을 닫는다
2. 나머지 참가자에게 "호스트 재접속 대기 중" 오버레이가 표시된다
3. 호스트가 같은 초대 링크로 재접속하고 동일한 이름으로 입장한다
4. 자동으로 호스트 권한이 복원되고 모든 참가자에게 알려진다
5. 대기 오버레이가 사라지고 게임이 재개된다

### 왜 이렇게 동작하는가
- **이름 완전 일치**: 별도의 인증 없이 이름만으로 호스트를 식별하는 간단한 방식이다. 같은 이름의 다른 사람이 먼저 입장하면 호스트가 될 수 있는 한계가 있다
- **500ms 지연**: P2P DataChannel이 완전히 열리기 전에 메시지가 유실되는 것을 방지하는 안전 마진이다
- **무기한 대기**: 호스트 재접속 타임아웃 없이 무기한 대기하되, 참가자가 원하면 홈으로 이탈할 수 있다

### 주요 정책
- POL-033: 호스트 이탈 시 무기한 대기 오버레이 표시 (상세: POLICIES.md)
- POL-034: 이름 완전 일치로 호스트 권한 자동 복원 (상세: POLICIES.md)
- POL-035: 호스트 복원 브로드캐스트 500ms 지연 전송 (상세: POLICIES.md)

### 관련 용어
- 호스트 재접속: 이름 매칭으로 호스트 권한을 복원하는 기능 (상세: GLOSSARY.md)

## P2P 연결 및 릴레이

### 이 기능은 무엇인가
참가자 간 게임 메시지를 전달하기 위한 통신 경로를 수립하는 기능이다. 우선 P2P 직접 연결을 시도하고, 실패하면 서버 릴레이로 자동 전환한다. 신규 참가자는 입장 시 기존 게임 상태를 동기화받는다.

### 사용자 시나리오
1. 방에 입장하면 다른 참가자와 P2P 연결을 시도한다
2. 8초 이내에 연결되면 P2P 모드로 동작한다
3. 연결에 실패하면 서버 릴레이 모드로 자동 전환된다
4. 신규 참가자는 입장 직후 현재 게임 상태를 동기화받는다

### 왜 이렇게 동작하는가
- **릴레이 자동 전환**: 방화벽이나 NAT 환경에서 P2P가 불가능해도 게임이 진행된다
- **신규 참가자만 동기화 요청**: 기존 참가자가 요청하면 빈 상태 응답이 돌아올 수 있다
- **자신의 투표 유지**: 동기화 시 자신의 투표 상태가 덮어씌워지지 않는다

### 주요 정책
- POL-036: P2P 실패 시 서버 릴레이 자동 전환, 비가역 (상세: POLICIES.md)
- POL-037: P2P 연결 폴백 대기 시간 8초 (상세: POLICIES.md)
- POL-038: 신규 참가자만 상태 동기화 요청 (상세: POLICIES.md)
- POL-039: 동기화 수신 시 자신의 투표 상태 유지 (상세: POLICIES.md)

### 관련 용어
- P2P 모드: WebRTC DataChannel 직접 전송 방식 (상세: GLOSSARY.md)
- 릴레이 모드: 서버 경유 메시지 전달 방식 (상세: GLOSSARY.md)
- 동기화 상태: 신규 참가자가 받는 전체 게임 상태 페이로드 (상세: GLOSSARY.md)
