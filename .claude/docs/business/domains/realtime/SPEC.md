# 실시간 통신 도메인 - 기획서

> 최종 갱신: 2026-03-25

## 도메인 개요

실시간 통신 도메인은 Planning Poker 세션에 참여한 사용자들이 서로의 투표 상태, 결과, 방 이벤트를 실시간으로 주고받을 수 있도록 하는 통신 기반을 담당한다. 참가자 간 직접 연결(P2P)을 우선 시도하되, 기업 보안 환경 등으로 직접 연결이 불가능한 경우 서버를 경유하는 중계 방식으로 자동 전환하여 어떤 네트워크 환경에서도 서비스가 정상 동작하도록 보장한다.

---

## SSE 시그널링 연결

### 이 기능은 무엇인가

사용자가 방에 입장하면, 브라우저와 서버 사이에 실시간 알림 채널이 자동으로 열린다. 이 채널을 통해 "누가 방에 있는지", "새로운 참가자가 들어왔는지", "누군가 나갔는지" 등의 이벤트를 즉시 수신한다. 이 알림 채널은 이후 참가자 간 직접 연결을 맺기 위한 준비 단계 역할을 한다.

### 사용자 시나리오

1. 사용자가 방을 생성하거나 초대 링크를 통해 방에 입장한다.
2. 시스템이 자동으로 서버와의 실시간 알림 채널을 연다.
3. 서버가 현재 방에 있는 참가자 목록을 즉시 전달한다.
4. 이후 새로운 참가자가 들어오거나 기존 참가자가 나가면 실시간으로 알림을 받는다.
5. 서버는 15초마다 연결 상태를 확인하는 신호를 보내어, 연결이 끊긴 참가자를 빠르게 감지한다.

### 왜 이렇게 동작하는가

- **방 입장 후 자동 연결**: 사용자가 별도 조작 없이 방에 들어가는 것만으로 실시간 통신이 시작된다. 닉네임과 식별 정보가 확정된 후에만 연결을 시작하여 불완전한 상태의 참가자가 목록에 표시되는 것을 방지한다.
- **15초 주기 상태 확인**: 탭을 닫거나 네트워크가 끊긴 참가자를 빠르게 감지하기 위함이다. 주기가 너무 짧으면 서버 부하가 증가하고, 너무 길면 이미 나간 참가자가 목록에 오래 남는 문제가 생긴다. 15초는 이 둘의 균형점이다.
- **중복 진입 이벤트 무시**: 네트워크 지연으로 같은 참가자의 입장 알림이 여러 번 올 수 있는데, 이미 목록에 있는 참가자는 무시하여 중복 표시를 방지한다.

### 주요 정책

- POL-001: 알림 채널 연결 시 참가자 식별 정보(ID, 이름)가 반드시 포함되어야 한다 (상세: POLICIES.md)
- POL-002: 방 생성 또는 참가가 완료된 상태에서만 알림 채널을 연다 (상세: POLICIES.md)
- POL-003: 혼자 있는 방에서는 직접 연결 협상을 시도하지 않는다 (상세: POLICIES.md)
- POL-004: 이미 목록에 있는 참가자의 중복 입장 이벤트는 무시한다 (상세: POLICIES.md)
- POL-005: 서버는 15초 간격으로 연결 상태 확인 신호를 전송한다 (상세: POLICIES.md)

### 에러 상황

- ERR-001: 연결 파라미터 누락 — 정상적인 방 입장 흐름을 통해 다시 시도하면 해결된다

### 관련 용어

- SSE (Server-Sent Events): 서버에서 브라우저로 실시간 알림을 보내는 단방향 통신 방식 (상세: GLOSSARY.md)
- Heartbeat: 연결이 살아 있는지 확인하기 위해 주기적으로 보내는 신호 (상세: GLOSSARY.md)
- 시그널링 (Signaling): 참가자 간 직접 연결을 맺기 전에 서로의 연결 정보를 교환하는 준비 과정 (상세: GLOSSARY.md)

## WebRTC P2P 협상

### 이 기능은 무엇인가

방에 2명 이상의 참가자가 있으면, 참가자들 사이에 서버를 거치지 않는 직접 연결(P2P)을 자동으로 맺는다. 새로 입장한 참가자가 기존 참가자 각각에게 "연결 요청"을 보내고, 상대가 "연결 수락"으로 응답하면 양방향 직접 통신이 가능해진다. 이 과정에서 연결 경로 후보들이 자동으로 탐색되어 최적의 통신 경로가 선택된다.

### 사용자 시나리오

1. 두 번째 참가자가 방에 입장한다.
2. 시스템이 자동으로 기존 참가자와의 직접 연결 협상을 시작한다.
3. 양측 브라우저가 연결 정보를 교환하고, 통신 가능한 네트워크 경로를 자동 탐색한다.
4. 최적의 경로가 발견되면 직접 연결이 성립하여, 이후 투표 데이터 등이 서버를 거치지 않고 참가자 간에 직접 전달된다.
5. 세 번째 이후 참가자도 동일한 과정으로 기존 모든 참가자와 각각 직접 연결을 맺는다.

### 왜 이렇게 동작하는가

- **직접 연결(P2P) 우선**: 서버를 거치지 않으므로 지연이 적고, 서버 부하가 없다. Planning Poker의 투표 데이터는 가볍기 때문에 참가자 간 직접 전달이 가장 효율적이다.
- **연결 경로 후보 묶음 전송 (100ms 윈도우)**: 연결 가능한 네트워크 경로 후보가 매우 짧은 간격으로 여러 개 발견되는데, 이를 하나씩 보내면 서버 왕복 횟수가 불필요하게 많아진다. 0.1초 이내의 후보들을 묶어 한 번에 보내 효율을 높인다.
- **후보 버퍼링**: 연결 경로 후보가 연결 정보 교환보다 먼저 도착하는 타이밍 문제가 있을 수 있다. 이 경우 후보를 임시로 보관했다가 연결 정보 교환이 완료된 후 일괄 적용하여 오류를 방지한다.
- **연결 요청 선도착 방어**: 네트워크 상황에 따라 "입장 알림"보다 "연결 요청"이 먼저 도착할 수 있다. 이런 경우에도 연결을 정상적으로 처리할 수 있도록 방어 로직이 적용되어 있다.

### 주요 정책

- POL-006: 서버 중계 모드가 활성화된 상태에서는 새 직접 연결을 시도하지 않는다 (상세: POLICIES.md)
- POL-007: 연결 정보 교환 완료 전 수신된 경로 후보는 임시 보관 후 일괄 적용한다 (상세: POLICIES.md)
- POL-008: 입장 알림보다 연결 요청이 먼저 도착해도 정상 처리한다 (상세: POLICIES.md)
- POL-009: 연결 시도가 실패(failed)하면 즉시 해당 참가자를 이탈 처리한다 (상세: POLICIES.md)
- POL-010: 연결 상태가 비정상(끊김/실패/종료)이면 해당 참가자를 이탈 처리한다 (상세: POLICIES.md)
- POL-011: 연결 경로 후보는 0.1초 이내 것을 묶어 한 번에 전송한다 (상세: POLICIES.md)

### 에러 상황

- ERR-002: 연결 요청 생성 실패 — 8초 후 서버 중계 모드로 자동 전환되어 서비스는 정상 이용 가능하다
- ERR-003: 연결 협상 오류 — 자동으로 서버 중계 모드로 전환된다. 지속 발생 시 네트워크 환경을 확인한다
- ERR-004: 연결 경로 후보 적용 실패 — 대부분 다른 경로를 통해 자동 연결되거나, 서버 중계 모드로 전환된다

### 관련 용어

- P2P (Peer-to-Peer): 서버를 거치지 않고 참가자 간 직접 데이터를 주고받는 통신 방식 (상세: GLOSSARY.md)
- SDP (Session Description Protocol): 참가자 간 직접 연결을 맺기 위해 교환하는 연결 정보 (상세: GLOSSARY.md)
- ICE candidate: 직접 연결에 사용할 수 있는 네트워크 경로 후보 (상세: GLOSSARY.md)
- Mesh Network: 모든 참가자가 서로 직접 연결되는 네트워크 구조 (상세: GLOSSARY.md)

## 서버 릴레이 폴백

### 이 기능은 무엇인가

기업 보안 환경(방화벽, 프록시 등)에서는 참가자 간 직접 연결이 차단될 수 있다. 이 기능은 직접 연결 시도 후 8초 이내에 연결이 성립되지 않으면, 자동으로 서버를 경유하는 중계 방식으로 전환한다. 중계 모드에서도 투표, 결과 공유 등 모든 기능이 동일하게 동작하며, 사용자는 화면 상단의 "서버 중계 모드" 표시로 현재 통신 방식을 확인할 수 있다.

### 사용자 시나리오

1. 참가자가 방에 입장하고 직접 연결 협상이 시작된다.
2. 기업 방화벽 등의 이유로 직접 연결이 차단되어 8초가 경과한다.
3. 시스템이 자동으로 서버 중계 모드로 전환한다.
4. 포커 테이블 화면에 "서버 중계 모드" 배지가 표시된다.
5. 이후 투표, 결과 공개 등 모든 게임 메시지가 서버를 경유하여 전달되지만, 사용자가 체감하는 차이는 거의 없다.

### 왜 이렇게 동작하는가

- **8초 대기 후 자동 전환**: 직접 연결이 성공할 수 있는 충분한 시간을 주되, 불가능한 환경에서 사용자를 너무 오래 기다리게 하지 않는 균형점이다. 너무 짧으면 느린 네트워크에서 불필요한 중계 전환이 발생하고, 너무 길면 기업 환경 사용자의 대기 시간이 길어진다.
- **한 번 전환되면 되돌리지 않음**: 직접 연결이 불가능했던 환경이 세션 중간에 갑자기 가능해질 가능성은 매우 낮다. 모드를 왔다 갔다 하면 오히려 불안정해질 수 있으므로, 한 번 중계 모드로 전환되면 세션이 끝날 때까지 유지한다.
- **투명한 전환**: 투표나 결과 공유를 호출하는 게임 로직은 통신 방식이 직접 연결인지 서버 중계인지 알 필요가 없다. 전환 시 게임 코드를 변경할 필요 없이 통신 경로만 자동으로 바뀐다.
- **추가 서버 불필요**: 기존 시그널링(연결 준비) 서버 인프라를 재사용하므로, 별도의 중계 서버를 구축하지 않아도 된다.
- **체감 성능 차이 미미**: Planning Poker는 "투표했다/결과 공개" 같은 매우 가벼운 텍스트만 주고받으므로, 서버를 경유해도 속도 차이를 느끼기 어렵다.

### 주요 정책

- POL-012: 첫 참가자 발견 후 8초 이내 직접 연결 실패 시 서버 중계 모드로 자동 전환한다 (상세: POLICIES.md)
- POL-013: 직접 연결이 성공하면 중계 전환 대기 타이머를 즉시 취소한다 (상세: POLICIES.md)
- POL-014: 중계 모드는 한 번 활성화되면 세션 내에서 해제되지 않는다 (상세: POLICIES.md)
- POL-015: 중계 전환 시 게임 상태 동기화 요청은 신규 참가자 측에서만 전송한다 (상세: POLICIES.md)
- POL-016: 중계 폴백 대기 시간은 8초이다 (상세: POLICIES.md)

### 에러 상황

- ERR-005: 서버 중계 모드 자동 전환 — 별도 조치 불필요. 자동으로 전환되어 모든 기능이 정상 동작한다

### 관련 용어

- 서버 릴레이 (Server Relay): 참가자 간 직접 연결 대신 서버를 경유하여 메시지를 전달하는 중계 방식 (상세: GLOSSARY.md)
- Transport Mode: 현재 통신 방식을 나타내는 상태 — "연결 중", "직접 연결(P2P)", "서버 중계" 중 하나 (상세: GLOSSARY.md)
- 폴백 (Fallback): 주요 방식이 실패했을 때 자동으로 대체 방식으로 전환하는 것 (상세: GLOSSARY.md)

## 피어 이탈 처리

### 이 기능은 무엇인가

참가자가 방을 나가거나 브라우저 탭을 닫거나 네트워크가 끊길 때, 해당 참가자를 참가자 목록에서 제거하고 나머지 참가자에게 이탈을 알린다. 능동적 이탈(나가기 버튼 클릭, 탭 닫기)과 비자발적 이탈(네트워크 단절, 연결 실패) 모두를 처리하며, 이탈 후 투표 완료 조건이 자동으로 재계산된다.

### 사용자 시나리오

**능동적 이탈 (나가기 클릭)**
1. 참가자가 "나가기" 버튼을 클릭한다.
2. 시스템이 다른 참가자들에게 이탈 메시지를 전송한다.
3. 나머지 참가자의 화면에서 해당 참가자가 즉시 사라진다.

**비자발적 이탈 (탭 닫기)**
1. 참가자가 브라우저 탭을 닫는다.
2. 브라우저가 닫히기 직전 서버에 이탈 신호를 전송한다.
3. 서버가 나머지 참가자에게 이탈을 알린다.

**비자발적 이탈 (네트워크 단절)**
1. 참가자의 네트워크가 끊긴다.
2. 서버의 실시간 알림 채널이 끊어지면서 해당 참가자의 이탈이 감지된다.
3. 서버가 나머지 참가자에게 이탈을 알린다.

**이탈 후 투표 영향**
1. 투표 진행 중 참가자가 이탈한다.
2. 시스템이 투표 완료 조건을 재계산한다 (예: 5명 중 3명 투표 완료 상태에서 미투표자 1명 이탈 → 4명 중 3명 투표로 변경).

### 왜 이렇게 동작하는가

- **이탈 경로 단일화**: 탭 닫기와 네트워크 단절 시 이탈 처리가 동시에 여러 경로로 발생할 수 있다. 이를 하나의 경로로 통합하여 같은 참가자의 이탈이 두 번 처리되는 것을 방지한다.
- **중복 처리 방지**: 서버에서 참가자 제거 시 "실제로 제거되었는지" 여부를 확인하고, 이미 제거된 참가자에 대해서는 추가 이탈 알림을 보내지 않는다.
- **빈 방 자동 정리**: 마지막 참가자가 나가면 방 자체를 서버 메모리에서 삭제하여 불필요한 리소스 점유를 방지한다.
- **즉시 반영**: 직접 연결 상태 감시를 통해 네트워크 단절을 빠르게 감지하고, 참가자 목록에 즉시 반영하여 "유령 참가자"가 오래 남는 것을 방지한다.

### 주요 정책

- POL-017: 동일 참가자에 대한 이탈 처리는 최초 1회만 실행한다 (상세: POLICIES.md)
- POL-018: 직접 연결 채널 종료 이벤트에서는 이탈 처리를 하지 않고, 서버 알림 경로만 사용한다 (상세: POLICIES.md)
- POL-019: 마지막 참가자가 나가면 방을 서버 메모리에서 즉시 삭제한다 (상세: POLICIES.md)

### 에러 상황

- 이 기능에서 정의된 별도 에러 시나리오는 없다. 이탈 처리는 정상 흐름의 일부로 동작한다.

### 관련 용어

- 피어 (Peer): 같은 방에 접속한 다른 참가자를 가리키는 용어 (상세: GLOSSARY.md)
- sendBeacon: 브라우저 탭을 닫을 때도 신뢰성 있게 서버에 신호를 보낼 수 있는 브라우저 기능 (상세: GLOSSARY.md)
- peer_left: 참가자가 방을 떠났음을 알리는 서버 이벤트 (상세: GLOSSARY.md)

## DataChannel 메시지 전송

### 이 기능은 무엇인가

투표, 결과 공개, 라운드 초기화, 방 종료, 참가자 추방 등 게임에서 발생하는 모든 실시간 메시지를 참가자들에게 전달하는 기능이다. 직접 연결(P2P) 모드에서는 참가자 간 직접 전달하고, 서버 중계 모드에서는 서버를 경유하여 전달한다. 어떤 모드이든 게임 로직은 동일한 방식으로 메시지를 보내고 받는다.

### 사용자 시나리오

**투표 메시지 전송**
1. 참가자가 카드를 선택하여 투표한다.
2. 시스템이 "투표 완료" 메시지를 모든 참가자에게 전송한다 (이 시점에서는 실제 투표값은 전달하지 않는다).
3. 다른 참가자의 화면에서 해당 참가자의 카드가 뒤집어진 상태(투표 완료)로 표시된다.

**결과 공개 메시지**
1. 모든 참가자가 투표를 완료한다.
2. 2초 카운트다운 후 실제 투표값이 포함된 "결과 공개" 메시지가 모든 참가자에게 전달된다.
3. 모든 참가자의 화면에서 실제 투표값과 통계(최빈값, 평균값)가 표시된다.

**신규 참가자 상태 동기화**
1. 신규 참가자가 진행 중인 세션에 입장한다.
2. 시스템이 자동으로 기존 참가자에게 현재 게임 상태를 요청한다.
3. 현재 티켓, 투표 현황, 참가자 목록 등 전체 게임 상태가 신규 참가자에게 전달되어 화면에 반영된다.

### 왜 이렇게 동작하는가

- **통신 방식 투명화**: 직접 연결이든 서버 중계든, 메시지를 보내는 코드는 동일하다. 통신 방식에 따른 분기는 내부적으로 자동 처리되므로 게임 로직이 단순하게 유지된다.
- **채널 상태 확인 후 전송**: 직접 연결 채널이 아직 준비되지 않았거나 이미 닫힌 경우에는 해당 채널을 건너뛰어 오류를 방지한다. 여러 참가자에게 보낼 때 한 명과의 채널에 문제가 있어도 나머지에게는 정상 전달된다.
- **손상된 메시지 무시**: 수신된 메시지가 올바른 형식이 아닌 경우 조용히 무시하여, 하나의 잘못된 메시지가 전체 게임 상태를 망가뜨리지 않도록 한다.
- **발신자 제외 전송**: 전체 참가자에게 메시지를 보낼 때 발신자 자신은 제외한다. 발신자는 이미 자신의 행동을 로컬에서 처리했으므로, 자신의 메시지를 다시 받으면 중복 처리가 발생하기 때문이다.

### 주요 정책

- POL-020: 직접 연결 채널이 열려 있는 경우에만 메시지를 전송한다 (상세: POLICIES.md)
- POL-021: 수신된 메시지의 형식이 올바르지 않으면 조용히 무시한다 (상세: POLICIES.md)
- POL-022: 전체 전송 시 메시지를 보낸 참가자 자신은 제외한다 (상세: POLICIES.md)

### 에러 상황

- 이 기능에서 정의된 별도 에러 시나리오는 없다. 채널 상태 확인과 형식 검증으로 오류가 사전에 방지된다.

### 관련 용어

- DataChannel: 참가자 간 직접 연결을 통해 데이터를 주고받는 통신 채널 (상세: GLOSSARY.md)
- broadcast: 메시지를 방의 모든 참가자에게 동시에 전달하는 전송 방식 (상세: GLOSSARY.md)
- DataMessage: 게임에서 주고받는 메시지의 총칭. 투표, 결과 공개, 초기화, 방 종료, 추방, 상태 동기화 등의 유형이 있다 (상세: GLOSSARY.md)
